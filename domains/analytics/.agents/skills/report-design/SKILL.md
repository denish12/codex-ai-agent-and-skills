# Дизайн отчёта — Проектирование макета аналитического отчёта

## Когда использовать
- При подготовке **финального аналитического отчёта** по результатам пайплайна (Сессия 5 Full Pipeline или финал Quick Pipeline).
- При необходимости **спроектировать структуру** PDF-отчёта: какие секции, какие визуализации, какая иерархия.
- При создании **report design specification** для передачи Верстальщику (Layouter).
- При проектировании **executive summary** на основе заключения Медиатора.
- При определении **цветовой палитры** и **визуальной иерархии** для печатного документа.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Заключение Медиатора | ✅ | Финальный вердикт, синтез, рекомендации |
| Отчёт Team Alpha | ✅ | Исследование, данные, фреймворки, стратегия |
| Отчёт Team Beta | ⬚ | Альтернативное исследование, критика, стратегия (Full Pipeline) |
| Interview Brief | ✅ | Тема, scope, ограничения, целевая аудитория |
| Appendix D (Synthesis) | ⬚ | Объединённые выводы (если одобрен) |
| Целевая аудитория | ✅ | CEO / инвестор / операционная команда / внешний клиент |
| Предпочтения по стилю | ⬚ | Корпоративный брендбук, цвета, шрифты |

## Источники данных
1. session-1-handoff.md — Interview Brief с scope и ограничениями.
2. session-2-handoff.md — Alpha Report с findings и фреймворками.
3. session-3-handoff.md — Beta Report с критикой и альтернативой.
4. session-4-handoff.md — Mediated Conclusion с финальным вердиктом.
5. Пользовательские предпочтения — аудитория, стиль, формат.

## Протокол

### Шаг 0 — Clarification
1. Уточнить целевую аудиторию отчёта.
2. Узнать о наличии корпоративного стиля / брендбука.
3. Определить уровень детализации (executive / detailed).
4. Уточнить формат (A4 portrait, количество страниц).
5. Зафиксировать предпочтения по визуализациям.

### Шаг 1 — Инвентаризация контента
1. Пройти все входные артефакты последовательно.
2. Для каждого элемента данных зафиксировать: тип, важность, рекомендуемое размещение.
3. Создать реестр контента: `[источник] × [элемент] × [тип данных] × [importance]`.
4. Отметить точки данных, которые требуют визуализации.
5. Отметить текстовые блоки для callout boxes и pull quotes.

### Шаг 2 — Структура отчёта
1. Определить секции: Title Page → TOC → Executive Summary → Main Section → Appendices.
2. Для Main Section: вердикт → метрики → стратегия → roadmap.
3. Для Appendices: A (Alpha) → B (Beta, Full Pipeline) → C (Mediator Reasoning) → D (Synthesis, опционально).
4. Назначить каждой секции количество страниц.
5. Определить page breaks между секциями.

### Шаг 3 — Выбор визуализаций
1. Для каждой числовой точки данных — выбрать тип графика (bar / line / pie / radar / scatter).
2. Для фреймворков (SWOT, BCG, Ansoff) — 2x2 CSS matrix.
3. Для процессов и потоков — Mermaid flowchart.
4. Для структур и иерархий — Mermaid mindmap.
5. Для позиционирования — Mermaid quadrant chart.
6. Для детальных сравнений — HTML table.
7. Максимум 3 визуализации на страницу.

### Шаг 4 — Визуальная иерархия
1. Определить 4 уровня emphasis: Hero → Key Finding → Supporting → Background.
2. Назначить severity indicators: Critical (красный), Warning (оранжевый), Positive (зелёный), Neutral (синий).
3. Определить использование callout boxes, pull quotes, severity badges.
4. Для каждой секции из структуры — назначить emphasis level.

### Шаг 5 — Цветовая палитра
1. Если есть брендбук — адаптировать палитру под корпоративные цвета.
2. Если нет — использовать стандартную профессиональную палитру (тёмно-синяя основа).
3. Определить цвета для: primary, secondary, accent, success, warning, critical, background, text, muted.
4. Определить палитру для Chart.js (6 серий).
5. Проверить контрастность WCAG AA и различимость в ч/б печати.

### Шаг 6 — Executive Summary
1. Написать черновик на основе заключения Медиатора.
2. Структура: контекст → методология → findings (3-5) → рекомендация → риски → следующие шаги.
3. Объём: максимум 1.5 страницы A4.
4. Язык: чёткий, без жаргона, адаптированный под аудиторию.

### Шаг 7 — Report Design Spec
1. Заполнить таблицу: `[секция] × [страницы] × [контент] × [визуализация] × [emphasis] × [примечания]`.
2. Приложить Visualization Map, Color Palette, Executive Summary Draft.
3. Передать спецификацию Верстальщику через HANDOFF.

## Валидация (Quality Gate)

- [ ] Каждая точка данных из анализа имеет назначенный тип визуализации
- [ ] Структура отчёта полная (Title → TOC → Summary → Main → Appendices)
- [ ] Executive summary не превышает 1.5 страницы
- [ ] Визуальная иерархия определена (4 уровня emphasis)
- [ ] Палитра подходит для печати (контрастность, ч/б различимость)
- [ ] Report Design Spec заполнен полностью
- [ ] Brief compliance: отчёт отвечает на вопросы из Interview Brief
- [ ] Максимум 3 визуализации на страницу
- [ ] Appendix D включён / исключён осознанно

## Handoff
Результат → Верстальщик (Layouter) для реализации в HTML/CSS/Chart.js.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Дизайн без инвентаризации | Пустые секции, лишние графики | Сначала пройти все артефакты |
| Один тип графика на всё | Неинформативно | Подбирать визуализацию под тип данных |
| Executive summary > 2 стр. | Руководитель не читает | Максимум 1.5 страницы |
| Палитра для экрана | В PDF серо и неразличимо | Проверять контрастность для печати |
| Перегруженные страницы | Каша из графиков | Максимум 3 визуализации на страницу |
| Изменение выводов | Нарушение chain of custody | Оформлять, не редактировать |
