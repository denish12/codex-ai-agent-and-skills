# Jobs To Be Done — Анализ задач клиента

## Когда использовать
- При разработке нового продукта — для выявления реальных задач, которые клиент пытается решить.
- При приоритизации фич — определить, какие задачи недообслужены и дают наибольшую возможность.
- При пересмотре value proposition — понять, за что клиент реально готов платить.
- При высоком churn — выяснить, какие задачи продукт не решает или решает плохо.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Продукт / сервис | ✅ | Что анализируем |
| Целевая аудитория | ✅ | Кто наш клиент (из `$icp-buyer-persona` или описание) |
| Контекст использования | ✅ | В какой ситуации клиент обращается к продукту |
| Существующие данные | ⬚ | Интервью, опросы, отзывы, тикеты поддержки |
| Конкуренты | ⬚ | Альтернативы, которые клиент использует сейчас |
| Гипотезы по Jobs | ⬚ | Предварительные предположения команды |

> Если обязательные поля не предоставлены — **запросить у пользователя** до начала анализа. Не додумывать.

## Источники данных
1. **Веб-поиск** — отзывы клиентов, обсуждения на форумах, видео-обзоры.
2. **Отзывы** — App Store, Google Play, G2, Capterra, Trustpilot, Amazon Reviews.
3. **Форумы и сообщества** — Reddit, Quora, Stack Overflow, тематические группы.
4. **Конкурентный анализ** — как конкуренты позиционируют свои решения (какие Jobs адресуют).
5. **Данные от пользователя** — интервью, опросы, CRM-данные, тикеты поддержки.

> Для каждого Job — минимум 1 источник, подтверждающий его существование.

## Протокол

### Шаг 0 — Сбор контекста
Проверить наличие обязательных входных данных. Определить уровень анализа: продукт целиком или конкретная функция / сценарий.

### Шаг 1 — Определение контекста (When / Where / Why)
- **When:** в какой ситуации возникает потребность (триггер).
- **Where:** физический и цифровой контекст использования.
- **Why:** какого прогресса клиент хочет достичь.
- Сформулировать контекстное утверждение: «Когда [ситуация], я хочу [прогресс], чтобы [результат]».

### Шаг 2 — Функциональные задачи (Functional Jobs)
Определить практические задачи, которые клиент пытается выполнить:
- Основная задача (Main Job) — ключевая цель.
- Связанные задачи (Related Jobs) — что ещё нужно сделать в контексте.
- Вспомогательные задачи (Consumption Chain Jobs) — покупка, настройка, обучение, обслуживание.
- Формулировка: глагол + объект + контекст (без привязки к решению).

### Шаг 3 — Эмоциональные задачи (Emotional Jobs)
Определить, как клиент хочет себя чувствовать:
- Уверенность, контроль, спокойствие.
- Удовлетворение, гордость, достижение.
- Избавление от тревоги, стресса, неопределённости.
- Формулировка: «Чувствовать [эмоция] при [контекст]».

### Шаг 4 — Социальные задачи (Social Jobs)
Определить, как клиент хочет выглядеть в глазах окружающих:
- Профессионализм, экспертность, компетентность.
- Инновационность, прогрессивность.
- Надёжность, ответственность.
- Формулировка: «Выглядеть [качество] в глазах [аудитория]».

### Шаг 5 — Приоритизация (Importance vs Satisfaction)
Для каждого Job оценить по двум шкалам (1-10):
- **Importance** — насколько важна эта задача для клиента.
- **Satisfaction** — насколько хорошо текущие решения справляются.
- Нанести на матрицу: высокая важность + низкая удовлетворённость = главная возможность.

### Шаг 6 — Outcome Expectations
Для каждого приоритетного Job определить ожидаемые результаты:
- Что клиент считает успешным выполнением задачи.
- Метрики успеха глазами клиента (скорость, качество, стоимость, удобство).
- Минимально приемлемый уровень и идеальный уровень.

### Шаг 7 — Конкурирующие решения
Для каждого Job определить, как клиент решает задачу сейчас:
- Прямые конкуренты (аналогичные продукты).
- Косвенные конкуренты (другие категории решений).
- DIY-решения (Excel, ручные процессы, workaround'ы).
- «Ничего не делать» (терпеть проблему).

### Шаг 8 — Opportunity Score
Рассчитать Opportunity Score для каждого Job:
- **Формула:** Opportunity = Importance + max(Importance - Satisfaction, 0).
- Jobs с Opportunity > 15 — высокий потенциал (underserved).
- Jobs с Opportunity < 10 — низкий потенциал (overserved или не важен).
- Ранжировать все Jobs по Opportunity Score.

## Валидация (Quality Gate)

- [ ] Определены минимум 5 функциональных задач с правильной формулировкой
- [ ] Определены минимум 3 эмоциональные задачи
- [ ] Определены минимум 2 социальные задачи
- [ ] Все задачи сформулированы без привязки к конкретному решению
- [ ] Проведена приоритизация по матрице Importance × Satisfaction
- [ ] Рассчитан Opportunity Score для всех задач
- [ ] Для каждой приоритетной задачи определены конкурирующие решения
- [ ] Outcome Expectations конкретны и измеримы
- [ ] Источники данных указаны для ключевых задач

> Если валидация не пройдена — доработать до повторного прохождения, а не передавать дальше.

## Handoff
Результат `$jtbd-analysis` является входом для:
- **Стратег / Медиатор** — приоритетные Jobs для стратегии продукта.
- `$icp-buyer-persona` — обогащение персон задачами и ожиданиями.
- `$competitive-analysis` — конкурирующие решения для бенчмаркинга.
- `$customer-journey-mapping` — задачи на каждом этапе пути клиента.
- `$unit-economics` — willingness to pay на основе ценности решения задач.

При передаче — использовать `$handoff` с ранжированным списком Jobs и Opportunity Scores.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Job привязан к решению | «Хочет использовать наше API» — это не Job | Формулировать через прогресс, не через продукт |
| Только функциональные Jobs | Эмоции и социальный контекст игнорируются | Всегда анализировать все три типа |
| Без приоритизации | Все Jobs кажутся одинаково важными | Использовать матрицу Importance × Satisfaction |
| Без конкурирующих решений | Непонятно, с чем конкурируем | Для каждого Job — текущие альтернативы |
| Слишком абстрактные Jobs | «Хочет быть успешным» — не actionable | Конкретный контекст, конкретный прогресс |
| Смешение Job и Feature | Job — задача клиента, Feature — решение | Разделять уровни анализа |
| Без Outcome Expectations | Непонятно, как измерить успех | Определить метрики успеха для каждого Job |
