# Gates — Контроль качества гейтов аналитического пайплайна

## Когда использовать
- **Перед каждым переходом** между гейтами — проверка deliverable текущего гейта.
- **При возврате на доработку** — повторная проверка после исправлений.
- **При завершении сессии** — проверка полноты материалов перед межсессионным `$handoff`.
- **При споре о готовности** — формальный арбитр: что сделано, а что нет.

> **Разграничение с `$board`:** `$gates` проверяет **качество и полноту** deliverable. `$board` отслеживает **статус и прогресс**. Gates отвечает на «готово ли?», Board — на «где мы сейчас?».

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Название гейта | ✅ | Какой гейт проверяем (с ID: COND-01, INT-01 и т.д.) |
| Режим пайплайна | ✅ | Full (`/analyze`) / Quick (`/quick-insight`) — определяет набор гейтов и критерии |
| Deliverables гейта | ✅ | Артефакты, которые агент передаёт на проверку |
| Итерация | ⬚ | Если повторная проверка — номер итерации + gap'ы предыдущей |
| Номер сессии | ⬚ | Текущая сессия (для Full Pipeline) |

## Критерии готовности по гейтам

### Conductor (COND-xx)
- [ ] Режим пайплайна определён и обоснован (Full / Quick)
- [ ] Scope задачи сформулирован (что анализируем, для чего, ограничения)
- [ ] Доска создана через `$board` с корректными ID гейтов
- [ ] Первый гейт активирован
- [ ] Для продолжающихся сессий: предыдущий handoff загружен и верифицирован

### Interviewer (INT-xx)
- [ ] Контекст пользователя собран (бизнес-вопрос, отрасль, размер компании, цели)
- [ ] Режим оценки выбран: scoring (количественный) или qualitative (качественный)
- [ ] Research brief сформирован с конкретными вопросами для исследования
- [ ] Определены ключевые метрики / критерии оценки
- [ ] Scope зафиксирован — что входит и что НЕ входит в анализ

### Researcher (RES-xx)
- [ ] Данные собраны по всем вопросам research brief
- [ ] Веб-поиск выполнен через `$web-research` (обязательно)
- [ ] Источники верифицированы: каждая точка данных имеет URL, дату, уровень достоверности
- [ ] Минимум источников достигнут (задаётся в research brief, по умолчанию >= 3 на тезис)
- [ ] Перекрёстная проверка: 2+ источника на ключевые данные
- [ ] Конфликтующие данные помечены и объяснены
- [ ] Данные актуальны: рыночные — не старше 12 месяцев, регуляторные — текущая версия

### Data Analyst (AN-xx)
- [ ] Фреймворки выбраны обоснованно (не менее 2 для Full Pipeline)
- [ ] Каждый фреймворк применён корректно — все секции заполнены
- [ ] Методология задокументирована: почему этот фреймворк, какие данные использованы
- [ ] Данные из Researcher привязаны к выводам (не «по нашему мнению», а «по данным [источник]»)
- [ ] Паттерны и инсайты выделены явно

### Strategist (ST-xx)
- [ ] Рекомендации actionable — конкретные действия, а не общие советы
- [ ] Каждая рекомендация привязана к evidence из Data Analyst / Researcher
- [ ] Приоритизация рекомендаций (impact / effort или аналогичная матрица)
- [ ] Риски обозначены для каждой рекомендации
- [ ] Временные горизонты указаны (краткосрочные / среднесрочные / долгосрочные)

### Mediator (MED-xx)
- [ ] Результаты обеих команд получены (для Full Pipeline) или единственной команды (Quick)
- [ ] Scoring или qualitative оценка выполнена (по выбранному в INT-01 режиму)
- [ ] Для scoring: критерии и веса определены, баллы обоснованы
- [ ] Для qualitative: аргументы обеих сторон сопоставлены, конфликты разрешены
- [ ] Финальное заключение сформулировано с обоснованием
- [ ] Сильные стороны каждой команды выделены

### Designer (DS-xx)
- [ ] Структура отчёта утверждена (оглавление, секции, порядок)
- [ ] Спецификации визуализаций определены (типы графиков, данные, палитра)
- [ ] Макет согласован с содержанием Mediator
- [ ] Брендинг / стилистика определены

### Layouter (LY-xx)
- [ ] HTML сгенерирован по макету Designer
- [ ] Все визуализации отрендерены
- [ ] Форматирование валидировано (отступы, шрифты, таблицы)
- [ ] PDF сгенерирован и проверен

### Release Gate (RG-xx)
- [ ] Все предыдущие гейты пройдены (статус `[✓]` на `$board`)
- [ ] PDF отчёт сгенерирован и проверен
- [ ] Данные актуальны на дату отчёта
- [ ] User sign-off получен

## Severity критериев

| Severity | Определение | Влияние на решение |
|----------|-------------|-------------------|
| Blocker | Обязательный deliverable отсутствует или критически неполный | FAIL — возврат на доработку |
| Gap | Deliverable есть, но неполный или с замечаниями | CONDITIONAL PASS — можно передать, но gap'ы зафиксированы |
| Note | Мелкое замечание, не влияет на переход | PASS — замечание передаётся следующему гейту как контекст |

## Решения

| Решение | Условие | Действие |
|---------|---------|----------|
| **PASS** | 0 Blocker + 0 Gap | Запросить user Approved → `$handoff` → следующий гейт |
| **CONDITIONAL PASS** | 0 Blocker + 1-2 Gap | Зафиксировать gap'ы → Запросить user Approved с пометкой → `$handoff` с пометкой |
| **FAIL** | >= 1 Blocker | Вернуть агенту с конкретным списком gap'ов → Не запрашивать Approved |

> **Абсолютное правило:** Никогда не пропускать гейты, даже если задача кажется простой. FAIL — доработка — повторная проверка. Без исключений.

### Специальные правила для верификации данных
- Researcher-гейты (RES-xx): отсутствие `$web-research` — **автоматический Blocker**.
- Данные без источника — **Blocker** (не Gap, не Note).
- Данные старше 12 месяцев (рыночные) — **Gap** с пометкой «требует обновления».

## Протокол

### Шаг 0 — Загрузка контекста
1. Определить какой гейт проверяем (по ID из `$board`).
2. Определить режим пайплайна (Full / Quick) и номер сессии.
3. Загрузить критерии готовности для этого гейта из реестра.
4. Если повторная проверка — загрузить gap'ы предыдущей итерации.

### Шаг 1 — Проверка deliverables
1. Для каждого критерия готовности — проверить: выполнен / не выполнен / N/A.
2. Для каждого невыполненного — присвоить severity (Blocker / Gap / Note).
3. Для каждого невыполненного — описать конкретный gap: что именно не хватает / что исправить.

### Шаг 2 — Принятие решения
1. Подсчитать severity.
2. Применить правила решений (PASS / CONDITIONAL PASS / FAIL).
3. Добавить запись в лог проверок.

### Шаг 3 — Действие по результату
- **PASS:** показать пользователю → запросить Approved → `$handoff` → обновить `$board`.
- **CONDITIONAL PASS:** показать gap'ы → запросить Approved с подтверждением gap'ов → `$handoff` с пометкой → обновить `$board`.
- **FAIL:** показать gap'ы → вернуть агенту → обновить `$board` (статус `[↩] Возврат`).

## Валидация (Quality Gate)

Проверка гейта считается завершённой если:

- [ ] Все критерии готовности пройдены (выполнен / не выполнен / N/A — нет пустых)
- [ ] Для каждого невыполненного указаны: severity, описание gap'а, что исправить
- [ ] Решение соответствует правилам (не субъективное)
- [ ] Лог проверок обновлён
- [ ] При PASS/CONDITIONAL PASS — user Approved получен
- [ ] `$board` обновлён
- [ ] Для Researcher-гейтов: `$web-research` верифицирован

## Handoff
Результат `$gates` является входными данными для: `$handoff` (приложение gate check к конверту).

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Пропуск гейта «потому что просто» | Ошибки проходят незамеченными | Никогда не пропускать. Без исключений |
| PASS без проверки всех критериев | Незамеченные gap'ы | Каждый критерий = выполнен / не выполнен / N/A |
| «В целом ок» вместо конкретики | Агент не знает что исправлять | Конкретный gap: «источник X не содержит дату публикации» |
| FAIL без описания gap'ов | Агент возвращается и не знает что делать | Каждый FAIL = список конкретных доработок |
| Approved без user sign-off | Нарушение протокола пайплайна | Всегда явное Approved от пользователя |
| Данные Researcher без веб-поиска | Непроверенные данные в отчёте | `$web-research` — обязателен, его отсутствие = Blocker |
| Одинаковая важность всех критериев | Мелочь блокирует, критичное проходит | Severity: Blocker / Gap / Note |
| Повторная проверка без сверки с gap'ами | Gap мог быть не исправлен | Загрузить gap'ы предыдущей итерации, проверить каждый |
