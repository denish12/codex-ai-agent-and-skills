# Web Research — Обязательный веб-поиск и верификация данных

## Когда использовать
- **Обязателен** для всех Researcher-гейтов (RES-xx) — без исключений.
- При **любом утверждении**, основанном на внешних данных (рыночная статистика, регуляторные нормы, конкурентные данные).
- При **верификации** данных, предоставленных пользователем — даже если пользователь уверен в их точности.
- При **обнаружении устаревших данных** — обновление и актуализация.

> **Абсолютное правило:** Researcher НЕ МОЖЕТ завершить гейт без выполнения `$web-research`. Отсутствие веб-поиска — автоматический **Blocker** при проверке `$gates`.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Research brief | ✅ | Вопросы для исследования (из Interviewer) |
| Тема исследования | ✅ | Основная тема / отрасль / компания |
| Данные пользователя | ⬚ | Данные, предоставленные пользователем для верификации |
| Ограничения поиска | ⬚ | География, язык, период, исключения |
| Минимум источников | ⬚ | Минимальное количество источников на тезис (по умолчанию 3) |

### Связь с другими скилами
| Скил | Взаимодействие | Когда |
|------|----------------|-------|
| `$gates` (RES-xx) | Отсутствие web-research = автоматический Blocker | При gate check каждого RES-гейта |
| `$handoff` | Результаты web-research передаются Data Analyst через handoff | После завершения исследования |
| `tam-sam-som` | Рыночные данные: TAM, CAGR, доли рынка | Когда Data Analyst применяет TAM/SAM/SOM |
| `competitive-analysis` | Данные о конкурентах: фичи, цены, доли | Когда Data Analyst применяет конкурентный анализ |
| `pest-analysis` | Макроданные: регуляция, экономика, демография | Когда Data Analyst применяет PEST/PESTEL |
| `trend-analysis` | Тренды: adoption rates, инвестиции, прогнозы | Когда Data Analyst применяет анализ трендов |
| `cohort-analysis` | Отраслевые бенчмарки retention | Если нет реальных данных по когортам |
| `unit-economics` | Бенчмарки CAC, LTV, churn по отрасли | Если нет реальных unit economics |
| Все фреймворк-скилы | Любые внешние данные, требуемые фреймворком | Всегда, когда фреймворк ссылается на `$web-research` |

> `$web-research` — **сервисный скил**: он обслуживает все аналитические скилы, поставляя верифицированные данные.

## Тирование источников

| Tier | Тип источника | Примеры | Надёжность | Использование |
|:----:|---------------|---------|:----------:|---------------|
| **T1** | Официальные отчёты, госстатистика, рецензируемые | Statista, Gartner, McKinsey, World Bank, Росстат, SEC filings | Высокая | Основа для Verified. 1 источник T1 = достаточно для Estimated |
| **T2** | Отраслевые медиа, аналитика, корпоративные блоги | TechCrunch, CB Insights, HH.ru Research, блоги компаний | Средняя | Подтверждение T1 или основа для Estimated. 2× T2 = Verified |
| **T3** | Форумы, соцсети, личные мнения, агрегаторы | Reddit, HN, Quora, блоги экспертов, Telegram-каналы | Низкая | Только для сигналов и гипотез. Не может быть Verified. Максимум Assumed |

**Правила:**
- Verified = 1× T1 + 1× T1/T2, или 3× T2 с совпадением данных.
- Estimated = 1× T1, или 2× T2.
- Assumed = T3, или косвенный расчёт, или экстраполяция.
- **T3 никогда не считается за Verified или Estimated.**

## Уровни достоверности данных

| Уровень | Маркер | Определение | Требования | В отчёте |
|---------|:------:|-------------|------------|----------|
| **Verified** | ✅ | Подтверждено 2+ независимыми источниками (T1/T2) | URL обоих, даты, совпадение | Можно утверждать |
| **Estimated** | ⚠️ | Один T1/T2 источник или расчёт | URL, методология, оговорка | «По оценкам...» |
| **Assumed** | 🔮 | Предположение, косвенные данные, T3 | Обоснование, косвенные источники | «Предположительно...» |

## Требования к актуальности данных

| Тип данных | Максимальный возраст | Действие при устаревании |
|-----------|:--------------------:|--------------------------|
| Рыночная статистика | 12 месяцев | Искать свежие или помечать ⚠️ Estimated |
| Регуляторные / правовые | Текущая версия | Обязательно проверить актуальность |
| Финансовые показатели компаний | Последний отчётный период | Указать период отчёта |
| Технологические тренды | 6 месяцев | Искать самые свежие |
| Демографические данные | 24 месяца | Допустимо, если нет свежих |

## Протокол

### Шаг 0 — Планирование поиска
1. Изучить research brief — выделить ключевые вопросы.
2. Для каждого вопроса определить минимум 3 поисковых запроса.
3. Включить год в запросы для обеспечения актуальности.
4. Запланировать запросы на контр-доказательства (counter-evidence).
5. Определить приоритет источников: T1 → T2 → T3.

### Шаг 1 — Выполнение поиска
1. Выполнить поисковые запросы через WebSearch.
2. Для каждого результата: извлечь данные через WebFetch.
3. Зафиксировать: URL, дату публикации, автора / организацию, **Tier источника**.
4. Минимум 3 запроса на каждую точку данных.

### Шаг 2 — Извлечение и структурирование данных
1. Из каждого источника извлечь конкретные данные (числа, факты, цитаты).
2. Привязать данные к вопросам из research brief.
3. Присвоить уровень достоверности (Verified / Estimated / Assumed) по правилам тирования.
4. Проставить дату-штамп для каждой точки данных.

### Шаг 3 — Перекрёстная проверка
1. Для каждой ключевой точки данных — найти подтверждение в 2+ источниках.
2. При расхождении данных: зафиксировать оба значения, указать источники, пометить как конфликт.
3. При конфликте — попытаться найти третий источник для разрешения.
4. Если конфликт не разрешён — передать обе версии с пометкой «Конфликт: [значение A] vs [значение B]».

### Шаг 4 — Поиск контр-доказательств
1. Для каждого ключевого тезиса — целенаправленно искать опровержения.
2. Сформулировать запросы вида: «[тезис] criticism», «[тезис] disadvantages», «[тезис] risks».
3. Зафиксировать найденные контр-аргументы.
4. Если контр-доказательств не найдено — зафиксировать как «контр-позиция не обнаружена».

### Шаг 5 — Формирование аудит-трейла
1. Записать все использованные поисковые запросы.
2. Записать все посещённые URL (даже если данные не использованы).
3. Записать причины отклонения источников (устарел, T3, нерелевантен, конфликтует).
4. Сформировать итоговую таблицу источников.

### Шаг 6 — Оценка качества исследования
Рассчитать метрики качества:

| Метрика | Формула | Значение | Порог |
|---------|---------|:--------:|:-----:|
| **Verified %** | Verified / Total × 100 | XX% | ≥ 50% — хорошо |
| **Coverage** | Вопросов покрыто / Всего вопросов × 100 | XX% | 100% — обязательно |
| **Source diversity** | Уникальных источников / Точек данных | X.X | ≥ 1.5 — хорошо |
| **Counter-evidence rate** | Тезисов с контр-поиском / Всего тезисов × 100 | XX% | 100% — обязательно |
| **Conflict rate** | Неразрешённых конфликтов / Всего точек × 100 | XX% | ≤ 10% — допустимо |

**Качество исследования:**

| Verified % | Оценка | Действие |
|:----------:|--------|----------|
| ≥ 70% | 🟢 Высокое | Можно передавать Data Analyst |
| 50-69% | 🟡 Среднее | Передавать с оговоркой, рекомендовать дополнительный поиск |
| < 50% | 🔴 Низкое | Доработать: найти больше T1/T2 источников |

### Шаг 7 — Компиляция результатов
1. Структурировать данные по вопросам research brief.
2. Для каждого вопроса: факты → источники (с Tier) → уровень достоверности → контр-позиция.
3. Выделить конфликты и неразрешённые вопросы.
4. Приложить метрики качества.
5. Сформировать выводы для передачи Data Analyst.

## Паттерны поисковых запросов

### Обязательные паттерны
| Паттерн | Пример | Цель |
|---------|--------|------|
| `[тема] [год]` | «SaaS market size 2025» | Актуальные данные |
| `[тема] statistics report` | «e-commerce Russia statistics report» | Официальные отчёты (T1) |
| `[тема] criticism risks` | «remote work criticism risks» | Контр-доказательства |
| `[компания] revenue financial results` | «Skillbox revenue 2025» | Финансовые данные |
| `[тема] vs [альтернатива]` | «Bitrix24 vs amoCRM 2025» | Сравнительные данные |
| `site:statista.com [тема]` | `site:statista.com cloud computing` | T1 источники |
| `[тема] research [отрасль]` | «retention benchmarks SaaS B2B» | Отраслевые бенчмарки |

### Запрещённые паттерны
| Паттерн | Почему | Альтернатива |
|---------|--------|-------------|
| Один общий запрос на всю тему | Поверхностные данные | Минимум 3 запроса на точку данных |
| Запрос без года | Устаревшие результаты | Всегда включать год |
| Только подтверждающие запросы | Confirmation bias | Обязательно искать контр-доказательства |
| Копирование данных без проверки | Недостоверные данные | Перекрёстная проверка 2+ источников |
| Только T3 источники | Низкая надёжность | Приоритет: T1 → T2 → T3 как дополнение |

## Пример — Web Research для «Рынок EdTech РФ, корпоративное обучение»

**Research brief (из INT-01):** 4 из 12 вопросов (фрагмент).

### Вопрос 1: Каков размер рынка EdTech РФ (TAM)?

| # | Факт | Источник | Tier | URL | Дата | Достоверность |
|---|------|----------|:----:|-----|------|:-------------:|
| 1 | EdTech РФ = $4.2B (2026) | Smart Ranking | T1 | smart-ranking.ru/... | 2026-01 | ✅ Verified |
| 2 | EdTech РФ = $3.9-4.5B (2026) | ВШЭ, ИСИЭЗ | T1 | hse.ru/... | 2025-11 | ✅ Verified |
| 3 | Корп. обучение = 43% от EdTech | ВШЭ | T1 | hse.ru/... | 2025-11 | ⚠️ Estimated |

**Контр-позиция:** Smart Ranking отмечает, что 30% рынка — теневой сектор (частные репетиторы), что может завышать TAM. Источник: smart-ranking.ru/methodology.
**Конфликты:** Нет (разброс $3.9-4.5B = нормальный для разных методологий).

### Вопрос 3: Какие технологические тренды меняют рынок?

| # | Факт | Источник | Tier | URL | Дата | Достоверность |
|---|------|----------|:----:|-----|------|:-------------:|
| 1 | AI-персонализация: adoption 12% в EdTech | Gartner | T1 | gartner.com/... | 2025-09 | ✅ Verified |
| 2 | AI-персонализация: Duolingo +40% retention | TechCrunch | T2 | techcrunch.com/... | 2025-07 | ✅ Verified (с #1) |
| 3 | LLM снижает стоимость курса ×5 | a16z | T2 | a16z.com/... | 2026-02 | ⚠️ Estimated |
| 4 | «AI в обучении переоценён для корп. сектора» | HBR | T1 | hbr.org/... | 2025-10 | ✅ Контр-позиция |

**Контр-позиция:** HBR аргументирует, что корп. обучение требует human-in-the-loop и AI-only решения не работают для soft skills.

### Метрики качества

| Метрика | Значение | Порог | Оценка |
|---------|:--------:|:-----:|:------:|
| Verified % | 62% (18/29) | ≥ 50% | 🟡 Среднее |
| Coverage | 100% (12/12 вопросов) | 100% | 🟢 |
| Source diversity | 1.7 (48 источников / 29 точек) | ≥ 1.5 | 🟢 |
| Counter-evidence rate | 100% (12/12 тезисов) | 100% | 🟢 |
| Conflict rate | 7% (2/29) | ≤ 10% | 🟢 |

**Качество: 🟡 Среднее** (Verified 62% — передавать с оговоркой, рекомендовать дополнительный T1 поиск для ⚠️ данных).

### Аудит-трейл (фрагмент)

| # | Запрос | Результатов | Использовано | Отклонено |
|---|--------|:-----------:|:------------:|:---------:|
| 1 | «EdTech market Russia 2025 2026» | 23 | 4 | 19 (устарели: 8, нерелевантны: 11) |
| 2 | «корпоративное обучение рынок РФ отчёт» | 15 | 3 | 12 (T3: 5, дубли: 4, устарели: 3) |
| 3 | «EdTech Russia criticism risks 2025» | 9 | 2 | 7 (нерелевантны) |

**Всего:** 38 запросов, 112 результатов, 48 источников использовано, 64 отклонено.

## Валидация (Quality Gate)

Веб-исследование считается завершённым если:

- [ ] Все вопросы research brief покрыты данными (Coverage = 100%)
- [ ] WebSearch / WebFetch использованы (не «по памяти»)
- [ ] Каждая точка данных: URL, дата, Tier источника, уровень достоверности
- [ ] Перекрёстная проверка выполнена (2+ источника на ключевые данные)
- [ ] Контр-доказательства искались целенаправленно для каждого тезиса (Counter-evidence = 100%)
- [ ] Данные актуальны по таблице требований
- [ ] Аудит-трейл заполнен (все запросы, URL, причины отклонения)
- [ ] Конфликтующие данные помечены и объяснены (Conflict rate ≤ 10%)
- [ ] Минимум источников достигнут (по умолчанию ≥ 3 на тезис)
- [ ] Verified % ≥ 50% (если < 50% — доработать или передать с оговоркой 🔴)
- [ ] T3 источники не использованы как Verified или Estimated

## Handoff
Результат `$web-research` является входными данными для:
- **Data Analyst (AN-xx)** — верифицированные данные для применения фреймворков.
- **Все фреймворк-скилы** — через Data Analyst: данные поступают в TAM/SAM/SOM, Competitive Analysis, PEST, SWOT и т.д.
- **`$gates` (RES-xx)** — gate check проверяет наличие и качество web-research.

Формат передачи: структурированный отчёт (по вопросам research brief) + таблица источников (с Tier и достоверностью) + метрики качества + аудит-трейл. При передаче — использовать `$handoff`.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Данные «по памяти» без поиска | Могут быть устаревшими или неточными | Всегда выполнять WebSearch |
| Один источник = «факт» | Нет перекрёстной проверки | 2+ источника (T1/T2) на ключевые данные |
| Нет контр-доказательств | Confirmation bias, однобокий анализ | Целенаправленно искать опровержения (100%) |
| URL без даты и Tier | Невозможно оценить актуальность и надёжность | Всегда: URL + дата + Tier (T1/T2/T3) |
| Все данные Verified | Нереалистично, скрывает неопределённость | Честно: Verified / Estimated / Assumed |
| Запросы без года | Устаревшие результаты | Включать текущий год в запрос |
| Пропуск аудит-трейла | Невозможно проверить качество исследования | Записывать все запросы, URL, причины отклонения |
| Только T3 источники | Низкая надёжность, не может быть Verified | Приоритет T1 → T2; T3 только как сигнал |
| Verified < 50% без доработки | Большинство данных ненадёжно | Доработать: найти T1/T2 или передать с оговоркой 🔴 |
