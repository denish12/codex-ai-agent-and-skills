# Board — Управление доской задач аналитического пайплайна

## Когда использовать
- При **инициализации** пайплайна — Conductor создаёт доску как первое действие.
- При **каждом переходе** между гейтами — совместно с `$handoff`.
- При **переходе между сессиями** — доска фиксирует текущий прогресс и передаётся через `$handoff`.
- При **блокере** — эскалация пользователю через протокол эскалации.
- При **запросе статуса** — пользователь спрашивает «где мы сейчас?».

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Режим пайплайна | ✅ | Full (`/analyze`, 5 сессий) / Quick (`/quick-insight`, 1 сессия) |
| Название задачи | ✅ | Краткое название аналитического запроса |
| ID задачи | ⬚ | Внешний ID (Linear, Notion, etc.), если есть |
| Ответственный | ⬚ | Кто инициировал запрос |
| Номер сессии | ⬚ | Текущая сессия (1-5 для Full Pipeline) |

> Если режим не определён — Conductor определяет по Decision Tree до создания доски.

## Идентификаторы гейтов

Каждый гейт получает уникальный ID по схеме `[РОЛЬ]-[NN]`:

| Префикс | Роль | Пример |
|----------|------|--------|
| `COND` | Conductor (Дирижёр) | COND-01 |
| `INT` | Interviewer (Интервьюер) | INT-01 |
| `RES` | Researcher (Исследователь) | RES-01, RES-02 |
| `AN` | Data Analyst (Аналитик данных) | AN-01, AN-02 |
| `ST` | Strategist (Стратег) | ST-01, ST-02 |
| `MED` | Mediator (Медиатор) | MED-01 |
| `DS` | Designer (Дизайнер) | DS-01 |
| `LY` | Layouter (Верстальщик) | LY-01 |

> Суффикс `-01` / `-02` используется для различения гейтов одного типа (например, Researcher Аналитиков vs Researcher Критиков).

## Статусы и State Machine

### Допустимые статусы

| Иконка | Статус | Значение |
|--------|--------|----------|
| `[ ]` | Ожидает | Гейт ещё не начат |
| `[→]` | В работе | Агент активно работает на этом гейте |
| `[✓]` | Завершён | Гейт пройден, deliverable зафиксирован, user sign-off получен |
| `[!]` | Блокер | Работа остановлена, требуется эскалация |
| `[↩]` | Возврат | Гейт отправлен на доработку (после review или user feedback) |
| `[⊘]` | Пропущен | Гейт пропущен по решению пользователя (только через протокол skip) |

### Допустимые переходы

```
[ ] Ожидает ──→ [→] В работе      (агент начал работу)
[ ] Ожидает ──→ [⊘] Пропущен     (user skip — только через протокол)
[→] В работе ──→ [✓] Завершён     (deliverable готов + user "Approved")
[→] В работе ──→ [!] Блокер       (обнаружен блокер)
[→] В работе ──→ [↩] Возврат      (user запросил доработку)
[!] Блокер ────→ [→] В работе      (блокер разрешён)
[↩] Возврат ───→ [→] В работе      (доработка начата)
[✓] Завершён ──→ [↩] Возврат      (поздний feedback, пересмотр)
```

> **Запрещённые переходы:**
> - `[ ] Ожидает` → `[✓] Завершён` (нельзя пропускать гейт без работы)
> - `[!] Блокер` → `[✓] Завершён` (блокер нужно сначала разрешить)
> - `[⊘] Пропущен` → любой статус (skip необратим)
> - Любой переход назад через несколько гейтов (возврат — только на текущий или предыдущий)

## Протокол

### Шаг 0 — Определение режима и сессии
1. Получить от пользователя тип запроса: `/analyze` (Full) или `/quick-insight` (Quick).
2. Для Full Pipeline — определить текущую сессию (1-5).
3. Если продолжение предыдущей сессии — загрузить доску из `$handoff` файла.

### Шаг 1 — Инициализация доски
1. Получить входные данные (режим, название, ID, сессия).
2. Определить набор гейтов по режиму и сессии (см. шаблоны ниже).
3. Создать доску со всеми гейтами в статусе `[ ] Ожидает`.
4. Перевести первый гейт (Conductor) в `[→] В работе`.
5. Добавить колонку «Сессия» для трекинга межсессионного прогресса.
6. Зафиксировать время старта.
7. Показать доску пользователю.

### Шаг 2 — Обновление при переходе
При каждом `$handoff`:
1. Текущий гейт → `[✓] Завершён`, заполнить deliverable и дату.
2. Следующий гейт → `[→] В работе`.
3. Рассчитать метрики прогресса (см. раздел «Метрики»).
4. Добавить запись в лог событий.
5. Показать обновлённую доску пользователю.

### Шаг 3 — Обработка блокера
1. Текущий гейт → `[!] Блокер`.
2. Определить severity блокера (см. раздел «Протокол эскалации»).
3. Заполнить описание блокера в лог событий.
4. Выполнить протокол эскалации.
5. После разрешения: гейт → `[→] В работе`, добавить запись в лог с решением.

### Шаг 4 — Обработка возврата
1. Гейт, на который идёт возврат → `[↩] Возврат`.
2. Все последующие гейты → `[ ] Ожидает` (сброс).
3. Зафиксировать причину возврата в лог событий.
4. После доработки: гейт → `[→] В работе`.

### Шаг 5 — Протокол Skip (пропуск гейта)
Используется когда пользователь запрашивает пропуск гейта.

1. **Запросить подтверждение:**
   ```
   ⚠️ Запрос на пропуск гейта [ID] [Название]

   **Влияние:** [На какие последующие гейты повлияет отсутствие deliverable этого гейта]
   **Риск:** [Что может пойти не так без этого гейта]

   Подтвердить пропуск? (Да / Нет)
   ```
2. Только при явном «Да» от пользователя → гейт переводится в `[⊘] Пропущен`.
3. В лог событий добавить: причина skip + подтверждение пользователя.
4. Следующий гейт → `[→] В работе`.

> **Нельзя пропускать:** COND-01 (инициализация), MED-01 (медиация — ядро пайплайна), LY-01 (финальный deliverable).

### Шаг 6 — Протокол Add (добавление гейта)
Используется когда в процессе работы выясняется, что нужен дополнительный гейт.

1. Conductor определяет: какой гейт, после какого существующего, обоснование.
2. **Запросить подтверждение:**
   ```
   ➕ Добавление гейта

   **Новый гейт:** [ID] [Название]
   **Позиция:** после [ID существующего гейта]
   **Обоснование:** [Почему необходим]
   **Влияние на сроки:** +[оценка времени]

   Подтвердить добавление? (Да / Нет)
   ```
3. При «Да» — вставить гейт в доску со статусом `[ ] Ожидает`, обновить нумерацию.
4. Зафиксировать в логе событий.

### Шаг 7 — Завершение сессии (межсессионный handoff)
1. Зафиксировать текущее состояние доски.
2. Сформировать пакет handoff:

   | Компонент | Описание |
   |-----------|----------|
   | Доска (полная) | Все гейты со статусами, deliverables, датами |
   | Лог событий | Полная история переходов, блокеров, решений |
   | Метрики | Прогресс, время на гейты, блокеры |
   | Контекст | Ключевые решения, открытые вопросы, зависимости |
   | Артефакты | Ссылки на все созданные deliverables |

3. Сохранить пакет через `$handoff` в файл `docs/analytics/session-N-handoff.md`.
4. Сгенерировать промпт для следующей сессии через `$session-prompt-generator`.

### Шаг 8 — Закрытие доски
1. Убедиться, что все гейты в `[✓] Завершён` или `[⊘] Пропущен`.
2. Рассчитать финальные метрики.
3. Добавить финальную запись в лог.
4. Показать итоговую доску + полный лог + метрики.

## Метрики прогресса

При каждом обновлении доски рассчитывать:

| Метрика | Формула | Пример |
|---------|---------|--------|
| Прогресс (%) | `завершённые / всего × 100` | 6/15 = 40% |
| Текущая сессия | `N из M` | 2 из 5 |
| Время на текущий гейт | `now - время_старта_гейта` | 12 мин |
| Среднее время на гейт | `Σ время_гейтов / кол-во_завершённых` | 8 мин |
| Блокеры (total) | `кол-во [!] за всё время` | 1 |
| Возвраты (total) | `кол-во [↩] за всё время` | 0 |
| ETA | `оставшиеся_гейты × среднее_время` | ~72 мин |

Отображать в строке прогресса под заголовком доски:
```
📊 Прогресс: 40% (6/15) | Сессия: 2/5 | Ср. время/гейт: 8 мин | ETA: ~72 мин | Блокеры: 1 | Возвраты: 0
```

## Протокол эскалации

### Severity блокеров

| Severity | Описание | Пример | Действие |
|:--------:|----------|--------|----------|
| 🔴 Critical | Пайплайн не может продолжаться, нет обходного пути | Отсутствуют ключевые данные, нужен ввод пользователя | Немедленная эскалация, остановка пайплайна |
| 🟡 Major | Гейт заблокирован, но можно работать на следующем параллельно | Один из источников недоступен, но есть альтернативные | Эскалация + предложение продолжить с оговоркой |
| 🟢 Minor | Качество deliverable снижается, но гейт проходим | Данные приблизительные, нет второго подтверждающего источника | Предупреждение в лог, пометка ⚠️ в deliverable |

### Шаблон эскалации

```
### ⚠️ Блокер — [Гейт ID] [Гейт]

**Задача:** [Название задачи]
**Сессия:** [N из M]
**Гейт:** [ID] [Название гейта]
**Severity:** [🔴 Critical / 🟡 Major / 🟢 Minor]
**Описание блокера:** [Что именно заблокировано и почему]
**Влияние:** [На какие последующие гейты влияет]
**Варианты решения:**
1. [Вариант A] — [плюсы / минусы]
2. [Вариант B] — [плюсы / минусы]

→ Ожидаю решение от пользователя.
```

## Интеграция с другими скилами

| Скил | Взаимодействие |
|------|----------------|
| `$handoff` | При каждом handoff — обновлять доску (шаг 2). При межсессионном handoff — сохранять полный пакет в файл (шаг 7) |
| `$gates` | Доска отражает гейты, определённые в `$gates`. Добавление гейтов (шаг 6) — согласовывать с `$gates` |
| `$session-prompt-generator` | Состояние доски + метрики + лог — входные данные для генерации промпта следующей сессии |
| `$web-research` | Researcher гейты включают обязательный веб-поиск |

## Валидация (Quality Gate)

Доска считается корректной если:

- [ ] Набор гейтов соответствует выбранному режиму пайплайна и сессии
- [ ] Ровно один гейт в статусе `[→] В работе` (или ноль, если все завершены / есть блокер)
- [ ] Нет пропущенных гейтов (`[✓]` / `[⊘]` идут подряд от начала)
- [ ] Все `[✓]` гейты имеют заполненный deliverable и дату
- [ ] Все `[⊘]` гейты имеют причину skip и подтверждение пользователя в логе
- [ ] Лог событий содержит запись для каждого перехода
- [ ] Нет запрещённых переходов
- [ ] Колонка «Сессия» заполнена для всех гейтов
- [ ] ID гейтов соответствуют схеме именования
- [ ] Метрики прогресса рассчитаны и отображены
- [ ] Блокеры имеют severity и разрешение зафиксировано в логе

## Handoff
Результат `$board` является входными данными для: `$handoff`, `$gates`, `$session-prompt-generator`.

Формат передачи при межсессионном handoff: доска + лог + метрики + контекст + артефакты (см. шаг 7).

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Пропуск обновления доски | Потеря трекинга, непонятно где мы | Обновлять при каждом `$handoff` |
| Несколько гейтов `[→]` одновременно | Параллельная работа без контроля | Только один `[→]` в любой момент |
| `[✓]` без deliverable | Непонятно, что было сделано | Всегда заполнять колонку «Deliverable» |
| Блокер без эскалации | Пользователь не знает о проблеме | Всегда использовать протокол эскалации с severity |
| Доска не показана пользователю | Пользователь вне контекста | Показывать доску при каждом обновлении |
| Ручная правка прошлых статусов | Искажение истории | Только `[↩] Возврат`, не перезапись |
| Потеря доски между сессиями | Следующая сессия начинает с нуля | Сохранять полный пакет через `$handoff` (шаг 7) |
| Отсутствие ID гейтов | Невозможно ссылаться на конкретный гейт | Всегда использовать ID: COND-01, INT-01 и т.д. |
| Пропуск гейта без протокола | Нарушение целостности пайплайна | Только через протокол skip (шаг 5) с подтверждением |
| Блокеры без severity | Все блокеры кажутся одинаковыми, нет приоритизации | Всегда определять 🔴/🟡/🟢 severity |
