# Board — Управление доской задач аналитического пайплайна

## Когда использовать
- При **инициализации** пайплайна — Conductor создаёт доску как первое действие.
- При **каждом переходе** между гейтами — совместно с `$handoff`.
- При **переходе между сессиями** — доска фиксирует текущий прогресс и передаётся через `$handoff`.
- При **блокере** — эскалация пользователю через протокол эскалации.
- При **запросе статуса** — пользователь спрашивает «где мы сейчас?».

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Режим пайплайна | ✅ | Full (`/analyze`, 5 сессий) / Quick (`/quick-insight`, 1 сессия) |
| Название задачи | ✅ | Краткое название аналитического запроса |
| ID задачи | ⬚ | Внешний ID (Linear, Notion, etc.), если есть |
| Ответственный | ⬚ | Кто инициировал запрос |
| Номер сессии | ⬚ | Текущая сессия (1-5 для Full Pipeline) |

> Если режим не определён — Conductor определяет по Decision Tree до создания доски.

## Идентификаторы гейтов

Каждый гейт получает уникальный ID по схеме `[РОЛЬ]-[NN]`:

| Префикс | Роль | Пример |
|----------|------|--------|
| `COND` | Conductor (Дирижёр) | COND-01 |
| `INT` | Interviewer (Интервьюер) | INT-01 |
| `RES` | Researcher (Исследователь) | RES-01, RES-02 |
| `AN` | Data Analyst (Аналитик данных) | AN-01, AN-02 |
| `ST` | Strategist (Стратег) | ST-01, ST-02 |
| `MED` | Mediator (Медиатор) | MED-01 |
| `DS` | Designer (Дизайнер) | DS-01 |
| `LY` | Layouter (Верстальщик) | LY-01 |

> Суффикс `-01` / `-02` используется для различения гейтов одного типа (например, Researcher Аналитиков vs Researcher Критиков).

## Статусы и State Machine

### Допустимые статусы

| Иконка | Статус | Значение |
|--------|--------|----------|
| `[ ]` | Ожидает | Гейт ещё не начат |
| `[→]` | В работе | Агент активно работает на этом гейте |
| `[✓]` | Завершён | Гейт пройден, deliverable зафиксирован, user sign-off получен |
| `[!]` | Блокер | Работа остановлена, требуется эскалация |
| `[↩]` | Возврат | Гейт отправлен на доработку (после review или user feedback) |

### Допустимые переходы

```
[ ] Ожидает ──→ [→] В работе      (агент начал работу)
[→] В работе ──→ [✓] Завершён     (deliverable готов + user "Approved")
[→] В работе ──→ [!] Блокер       (обнаружен блокер)
[→] В работе ──→ [↩] Возврат      (user запросил доработку)
[!] Блокер ────→ [→] В работе      (блокер разрешён)
[↩] Возврат ───→ [→] В работе      (доработка начата)
[✓] Завершён ──→ [↩] Возврат      (поздний feedback, пересмотр)
```

> **Запрещённые переходы:**
> - `[ ] Ожидает` → `[✓] Завершён` (нельзя пропускать гейт)
> - `[!] Блокер` → `[✓] Завершён` (блокер нужно сначала разрешить)
> - Любой переход назад через несколько гейтов (возврат — только на текущий или предыдущий)

## Протокол

### Шаг 0 — Определение режима и сессии
1. Получить от пользователя тип запроса: `/analyze` (Full) или `/quick-insight` (Quick).
2. Для Full Pipeline — определить текущую сессию (1-5).
3. Если продолжение предыдущей сессии — загрузить доску из `$handoff` файла.

### Шаг 1 — Инициализация доски
1. Получить входные данные (режим, название, ID, сессия).
2. Определить набор гейтов по режиму и сессии (см. шаблоны ниже).
3. Создать доску со всеми гейтами в статусе `[ ] Ожидает`.
4. Перевести первый гейт (Conductor) в `[→] В работе`.
5. Добавить колонку «Сессия» для трекинга межсессионного прогресса.
6. Показать доску пользователю.

### Шаг 2 — Обновление при переходе
При каждом `$handoff`:
1. Текущий гейт → `[✓] Завершён`, заполнить deliverable и дату.
2. Следующий гейт → `[→] В работе`.
3. Добавить запись в лог событий.
4. Показать обновлённую доску пользователю.

### Шаг 3 — Обработка блокера
1. Текущий гейт → `[!] Блокер`.
2. Заполнить описание блокера в лог событий.
3. Выполнить протокол эскалации (см. ниже).
4. После разрешения: гейт → `[→] В работе`, добавить запись в лог.

### Шаг 4 — Обработка возврата
1. Гейт, на который идёт возврат → `[↩] Возврат`.
2. Все последующие гейты → `[ ] Ожидает` (сброс).
3. Зафиксировать причину возврата в лог событий.
4. После доработки: гейт → `[→] В работе`.

### Шаг 5 — Завершение сессии (межсессионный handoff)
1. Зафиксировать текущее состояние доски.
2. Передать доску через `$handoff` в файл `docs/analytics/session-N-handoff.md`.
3. Сгенерировать промпт для следующей сессии через `$session-prompt-generator`.

### Шаг 6 — Закрытие доски
1. Убедиться, что все гейты в `[✓] Завершён`.
2. Добавить финальную запись в лог.
3. Показать итоговую доску + полный лог.

## Протокол эскалации

При переводе гейта в `[!] Блокер` — сформировать сообщение:

```
### ⚠️ Блокер — [Гейт ID] [Гейт]

**Задача:** [Название задачи]
**Сессия:** [N из M]
**Гейт:** [ID] [Название гейта]
**Описание блокера:** [Что именно заблокировано и почему]
**Влияние:** [На какие последующие гейты влияет]
**Варианты решения:**
1. [Вариант A] — [плюсы / минусы]
2. [Вариант B] — [плюсы / минусы]

→ Ожидаю решение от пользователя.
```

## Интеграция с другими скилами

| Скил | Взаимодействие |
|------|----------------|
| `$handoff` | При каждом handoff — обновлять доску (шаг 2). При межсессионном handoff — сохранять доску в файл |
| `$gates` | Доска отражает гейты, определённые в `$gates` |
| `$session-prompt-generator` | Состояние доски — входные данные для генерации промпта следующей сессии |
| `$web-research` | Researcher гейты включают обязательный веб-поиск |

## Валидация (Quality Gate)

Доска считается корректной если:

- [ ] Набор гейтов соответствует выбранному режиму пайплайна и сессии
- [ ] Ровно один гейт в статусе `[→] В работе` (или ноль, если все завершены / есть блокер)
- [ ] Нет пропущенных гейтов (`[✓]` идут подряд от начала)
- [ ] Все `[✓]` гейты имеют заполненный deliverable и дату
- [ ] Лог событий содержит запись для каждого перехода
- [ ] Нет запрещённых переходов
- [ ] Колонка «Сессия» заполнена для всех гейтов
- [ ] ID гейтов соответствуют схеме именования

## Handoff
Результат `$board` является входными данными для: `$handoff`, `$gates`, `$session-prompt-generator`.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Пропуск обновления доски | Потеря трекинга, непонятно где мы | Обновлять при каждом `$handoff` |
| Несколько гейтов `[→]` одновременно | Параллельная работа без контроля | Только один `[→]` в любой момент |
| `[✓]` без deliverable | Непонятно, что было сделано | Всегда заполнять колонку «Deliverable» |
| Блокер без эскалации | Пользователь не знает о проблеме | Всегда использовать протокол эскалации |
| Доска не показана пользователю | Пользователь вне контекста | Показывать доску при каждом обновлении |
| Ручная правка прошлых статусов | Искажение истории | Только `[↩] Возврат`, не перезапись |
| Потеря доски между сессиями | Следующая сессия начинает с нуля | Сохранять доску через `$handoff` в файл |
| Отсутствие ID гейтов | Невозможно ссылаться на конкретный гейт | Всегда использовать ID: COND-01, INT-01 и т.д. |
