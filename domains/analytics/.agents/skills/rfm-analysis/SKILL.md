# RFM-анализ — Сегментация клиентов по давности, частоте и сумме покупок

## Когда использовать
- При построении стратегии удержания клиентов — понять, кого удерживать, а кого реактивировать.
- При оптимизации маркетингового бюджета — направить ресурсы на наиболее ценные сегменты.
- При запуске персонализированных кампаний — разный месседж для разных сегментов.
- При анализе клиентской базы — выявление Champions, лояльных, рисковых и потерянных клиентов.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Бизнес / продукт | ✅ | Что анализируем |
| Бизнес-модель | ✅ | E-commerce / SaaS / Marketplace / Сервис |
| Данные транзакций | ⬚ | Таблица: клиент, дата, сумма (если есть реальные данные) |
| Период анализа | ⬚ | За какой период (по умолчанию — 12 месяцев) |
| Количество клиентов | ⬚ | Размер базы для калибровки шкалы |
| Средний чек | ⬚ | Для моделирования, если нет реальных данных |
| Частота покупок | ⬚ | Средняя частота для моделирования |

> Если обязательные поля не предоставлены — **запросить у пользователя** до начала анализа. Не додумывать.

> Если реальных данных нет — скил работает с **моделированными данными**, явно помеченными как гипотетические. Методология остаётся применимой при появлении реальных данных.

## Источники данных
1. **Транзакционные данные** — CRM, платёжная система, ERP (если предоставлены).
2. **Веб-поиск** — бенчмарки по отрасли (средний чек, частота покупок, retention rate).
3. **Отраслевые отчёты** — типичные RFM-распределения для данного типа бизнеса.
4. **Аналитика продукта** — данные активности пользователей (для SaaS — логины, действия).
5. **Данные от пользователя** — выгрузки из CRM, Google Analytics, платёжного провайдера.

> При работе с моделированными данными — указать источники бенчмарков. Данные старше 12 месяцев — пометить ⚠️.

### Связь с другими скилами
| Скил | Что берём | Когда вызывать |
|------|-----------|----------------|
| `web-research` | Отраслевые бенчмарки: средний чек, частота, retention | Если нет реальных данных — для моделирования |
| `cohort-analysis` | Retention curves для калибровки Recency-порогов (шаг 2) | Если есть когортные данные |
| `competitive-analysis` | Бенчмарки конкурентов по retention и LTV | Для сравнения с рыночными стандартами |
| `icp-buyer-persona` | Профиль Champions → обогащение ICP (шаг 8) | После сегментации: Champions = эталон ICP |
| `unit-economics` | LTV per segment для расчёта бюджета на реактивацию | Для оценки ROI по сегментам |
| `customer-journey-mapping` | Точки оттока для At Risk / About to Sleep | Для понимания, где именно теряем клиентов |

## Протокол

### Шаг 0 — Сбор контекста
1. Проверить наличие обязательных данных.
2. Определить режим: **реальные данные** или **моделированные**.
3. Зафиксировать период анализа.
4. Определить адаптацию под бизнес-модель:

   | Бизнес-модель | Recency = | Frequency = | Monetary = |
   |---------------|-----------|-------------|------------|
   | E-commerce | Дни с последнего заказа | Кол-во заказов | Общая сумма заказов |
   | SaaS (подписка) | Дни с последнего логина/действия | Кол-во активных месяцев | MRR или план подписки |
   | SaaS (usage-based) | Дни с последнего использования | Кол-во сессий/действий | Общая сумма оплат |
   | Marketplace | Дни с последней сделки | Кол-во сделок | GMV (объём сделок) |
   | Сервис (офлайн) | Дни с последнего визита | Кол-во визитов | Общая сумма чеков |

### Шаг 1 — Определение области данных
- Определить единицу анализа: клиент / аккаунт / подписчик.
- Определить «транзакцию» для данной бизнес-модели (см. таблицу шага 0).
- Определить период: начальная и конечная дата.
- Исключить аномалии: тестовые аккаунты, возвраты, аномальные суммы (>3σ).

### Шаг 2 — Recency Score (R)
1. Рассчитать дни с последней транзакции для каждого клиента.
2. Определить пороги на основе бизнес-цикла:

   | Score | E-commerce | SaaS (подписка) | Marketplace |
   |:-----:|:----------:|:---------------:|:-----------:|
   | R=5 | 0-30 дней | 0-7 дней | 0-14 дней |
   | R=4 | 31-90 | 8-30 | 15-45 |
   | R=3 | 91-180 | 31-60 | 46-90 |
   | R=2 | 181-365 | 61-120 | 91-180 |
   | R=1 | >365 | >120 | >180 |

   > Пороги — ориентир. Калибровать под реальное распределение (квинтили) если есть данные.

### Шаг 3 — Frequency Score (F)
1. Подсчитать количество транзакций на клиента за период.
2. Разбить на 5 групп (квинтили или бизнес-логика).
3. F=5: топ-20% по частоте. F=1: однократная / минимальная.

### Шаг 4 — Monetary Score (M)
1. Рассчитать общую сумму на клиента за период.
2. Разбить на 5 групп (квинтили или бизнес-логика).
3. M=5: топ-20% по сумме. M=1: минимальные траты.
4. Для SaaS с фиксированной подпиской: заменить M на **Engagement Score** (глубина использования: фичи, интеграции, кол-во пользователей в аккаунте).

### Шаг 5 — Сегментация
Присвоить каждому клиенту RFM-код (111-555) и сгруппировать:

| Сегмент | RFM-диапазон | Описание | Типичная доля |
|---------|:------------:|----------|:-------------:|
| **Champions** | R=5, F=4-5, M=4-5 | Лучшие клиенты, покупают часто и много | 5-10% |
| **Loyal Customers** | R=4-5, F=3-5, M=3-5 | Регулярные и ценные | 10-15% |
| **Potential Loyalists** | R=4-5, F=1-3, M=1-3 | Новые с потенциалом роста | 10-15% |
| **Recent Customers** | R=5, F=1, M=1 | Только пришли, первая покупка | 5-10% |
| **Promising** | R=3-4, F=1-2, M=1-2 | Есть интерес, нужна активация | 10-15% |
| **Need Attention** | R=3, F=3, M=3 | Средние, рискуют уйти | 5-10% |
| **About to Sleep** | R=2-3, F=1-2, M=1-2 | Теряют интерес | 10-15% |
| **At Risk** | R=1-2, F=3-5, M=3-5 | Были активны и ценны, уходят | 5-10% |
| **Cannot Lose** | R=1, F=5, M=5 | VIP, которые ушли — критично вернуть | 1-3% |
| **Hibernating** | R=1-2, F=1-2, M=1-2 | Давно неактивны, низкая ценность | 10-15% |
| **Lost** | R=1, F=1, M=1 | Потеряны | 5-10% |

### Шаг 6 — Оценка ценности сегментов
Для каждого сегмента рассчитать:

| Сегмент | % базы | % выручки | Ср. LTV | Revenue/клиент | Приоритет бюджета |
|---------|:------:|:---------:|:-------:|:--------------:|:-----------------:|
| Champions | X% | X% | $XXX | $XXX | Высокий (удержание) |
| At Risk | X% | X% | $XXX | $XXX | Высокий (реактивация) |
| Lost | X% | X% | $XXX | $XXX | Низкий (отпустить) |

**Правило Парето:** обычно Champions + Loyal = 15-25% базы, но 60-80% выручки. Если распределение другое — зафиксировать как аномалию.

### Шаг 7 — Стратегия и персонализация по сегментам

| Сегмент | Цель | Действие | Канал | Тон | Предложение | KPI |
|---------|------|----------|-------|-----|-------------|-----|
| Champions | Удержание + адвокация | Эксклюзивный доступ, реферальная программа | Email, personal | Благодарный, VIP | Ранний доступ, бонус за реферал | NPS, referral rate |
| Loyal | Upsell + лояльность | Кросс-продажи, программа лояльности | Email, in-app | Дружеский, экспертный | Upgrade, bundle | ARPU, retention |
| Potential Loyalists | Вовлечение + онбординг | Guided onboarding, серия обучающих писем | Email, in-app | Поддерживающий | Бонус за 2-ю покупку | F→2+, activation |
| Recent | Активация | Welcome-серия, aha-moment | Email, push | Приветливый | Промокод на 2-й заказ | Day-7 retention |
| At Risk | Реактивация | «Мы скучаем», специальное предложение | Email, SMS | Заботливый, срочный | Скидка 20%, персональный оффер | Win-back rate |
| Cannot Lose | Срочная реактивация | Звонок менеджера, персональное предложение | Звонок, email | Личный, VIP | Максимальная скидка, upgrade | Win-back rate |
| Lost | Win-back или отпустить | Финальное письмо, опрос причины ухода | Email | Нейтральный | Опрос + последний шанс | Churn reason data |

### Шаг 8 — Миграционный анализ
Отслеживать перемещение клиентов между сегментами:

1. Сравнить RFM-сегменты за текущий и предыдущий период:

   | Миграция | Количество | % | Интерпретация | Действие |
   |----------|:----------:|:-:|---------------|----------|
   | Champions → At Risk | [N] | X% | Потеря лучших клиентов — критично | Немедленная реактивация |
   | Potential Loyalists → Loyal | [N] | X% | Здоровый рост — онбординг работает | Масштабировать подход |
   | At Risk → Lost | [N] | X% | Реактивация не сработала | Пересмотреть стратегию |
   | Recent → Promising | [N] | X% | Активация работает | Продолжать |

2. Ключевые миграции для мониторинга:
   - **Негативные:** Champions/Loyal → At Risk, At Risk → Lost, Potential → Hibernating.
   - **Позитивные:** Recent → Potential → Loyal → Champions.
3. Рекомендуемая частота пересчёта:

   | Бизнес-модель | Частота |
   |---------------|---------|
   | E-commerce (FMCG) | Еженедельно |
   | E-commerce (дорогие товары) | Ежемесячно |
   | SaaS | Ежемесячно |
   | Marketplace | Раз в 2 недели |
   | Сервис (офлайн) | Ежеквартально |

## Пример — E-commerce: магазин спортивного питания

**Контекст:** Онлайн-магазин спортивного питания, B2C, 12 000 клиентов, период 12 мес. Средний чек $45, средняя частота 3.2 покупки/год.

### Шкала оценки

| Метрика | 5 | 4 | 3 | 2 | 1 |
|---------|:-:|:-:|:-:|:-:|:-:|
| **Recency** (дни) | 0-30 | 31-60 | 61-120 | 121-240 | >240 |
| **Frequency** (заказов) | ≥6 | 4-5 | 3 | 2 | 1 |
| **Monetary** ($) | >$300 | $200-300 | $100-200 | $50-100 | <$50 |

### Сегменты

| Сегмент | % базы | % выручки | Кол-во | Ср. LTV | Стратегия |
|---------|:------:|:---------:|:------:|:-------:|-----------|
| Champions | 8% | 35% | 960 | $520 | Реферальная программа, VIP-доступ |
| Loyal | 12% | 28% | 1440 | $340 | Upsell на подписку, программа лояльности |
| Potential Loyalists | 11% | 8% | 1320 | $105 | Онбординг: «3-й заказ = бесплатная доставка» |
| At Risk | 7% | 12% | 840 | $250 | «Мы скучаем» + скидка 15% на любимую категорию |
| Cannot Lose | 2% | 5% | 240 | $360 | Звонок менеджера, персональный оффер |
| Lost | 15% | 2% | 1800 | $25 | Финальный email + опрос причины ухода |

**Проверка Парето:** Champions + Loyal = 20% базы, 63% выручки ✅ (здоровое распределение).

### Миграционный анализ (мес. → мес.)

| Миграция | Кол-во | % | Сигнал |
|----------|:------:|:-:|--------|
| Champions → At Risk | 45 | 4.7% | 🔴 Потеря VIP — запустить win-back |
| Potential → Loyal | 180 | 13.6% | 🟢 Онбординг работает |
| At Risk → Lost | 120 | 14.3% | 🟡 Реактивация недостаточно эффективна |

**Рекомендация:** Champions → At Risk = 4.7% — выше бенчмарка (3%). Расследовать: цена? качество? конкурент?

## Валидация (Quality Gate)

- [ ] Определены все три метрики (R, F, M) с чёткими порогами, адаптированными под бизнес-модель
- [ ] Пороги квинтилей обоснованы бизнес-циклом (или квинтилями реальных данных)
- [ ] Выделены минимум 6 именованных сегментов с RFM-диапазонами
- [ ] Для каждого сегмента: стратегия, действие, канал, тон, KPI
- [ ] Ценность сегментов оценена: % базы, % выручки, средний LTV
- [ ] Парето-проверка: Champions + Loyal ≈ 15-25% базы, 60-80% выручки
- [ ] Миграционный анализ: ключевые переходы между сегментами зафиксированы
- [ ] Частота пересчёта рекомендована для данной бизнес-модели
- [ ] Если данные моделированные — явно указано с источниками бенчмарков
- [ ] Нет логических противоречий в сегментации (RFM-коды корректны)
- [ ] Источники данных указаны; данные старше 12 мес помечены ⚠️

> Если валидация не пройдена — доработать до повторного прохождения, а не передавать дальше.

## Handoff
Результат `$rfm-analysis` является входом для:
- **Стратег / Медиатор** — сегменты клиентов для стратегических решений.
- **`unit-economics`** — LTV по сегментам, retention rates, бюджет на реактивацию.
- **`cohort-analysis`** — группировка по когортам внутри сегментов.
- **`customer-journey-mapping`** — точки оттока для At Risk / About to Sleep.
- **`icp-buyer-persona`** — профиль Champions = эталон ICP.

Формат передачи: таблица сегментов (% базы, % выручки, LTV) + стратегии + миграционный анализ. При передаче — использовать `$handoff`.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Одинаковые пороги для всех бизнесов | У SaaS и e-commerce разные циклы покупки | Калибровать пороги под бизнес-модель (таблица шаг 0) |
| Моделированные данные без пометки | Решения на основе гипотез, а не фактов | Явно помечать, указывать источники бенчмарков |
| Слишком много сегментов (>11) | Невозможно создать уникальную стратегию для каждого | 6-11 именованных сегментов |
| Сегментация без стратегии | RFM ради RFM, нет actionable-выводов | Каждый сегмент = цель + действие + канал + KPI |
| Игнорирование M при подписке | В SaaS Monetary менее вариативна | Заменить M на Engagement Score для SaaS |
| Статичный анализ без миграции | Не видно, как клиенты перемещаются между сегментами | Миграционный анализ: негативные и позитивные переходы |
| Без персонализации | Одинаковое сообщение для Champions и At Risk | Разный тон, канал, предложение, частота по сегментам |
| Сегменты без ценности | Непонятно, куда направить бюджет | % базы + % выручки + LTV per segment + Парето-проверка |
| Бенчмарки без источника | Невозможно верифицировать | Каждый бенчмарк — с источником и датой |
