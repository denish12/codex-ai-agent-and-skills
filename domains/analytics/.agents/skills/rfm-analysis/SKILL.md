# RFM-анализ — Сегментация клиентов по давности, частоте и сумме покупок

## Когда использовать
- При построении стратегии удержания клиентов — понять, кого удерживать, а кого реактивировать.
- При оптимизации маркетингового бюджета — направить ресурсы на наиболее ценные сегменты.
- При запуске персонализированных кампаний — разный месседж для разных сегментов.
- При анализе клиентской базы — выявление Champions, лояльных, рисковых и потерянных клиентов.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Бизнес / продукт | ✅ | Что анализируем |
| Бизнес-модель | ✅ | E-commerce / SaaS / Marketplace / Сервис |
| Данные транзакций | ⬚ | Таблица: клиент, дата, сумма (если есть реальные данные) |
| Период анализа | ⬚ | За какой период (по умолчанию — 12 месяцев) |
| Количество клиентов | ⬚ | Размер базы для калибровки шкалы |
| Средний чек | ⬚ | Для моделирования, если нет реальных данных |
| Частота покупок | ⬚ | Средняя частота для моделирования |

> Если реальных данных нет — скил работает с **моделированными данными**, явно помеченными как гипотетические. Методология остаётся применимой при появлении реальных данных.

## Источники данных
1. **Транзакционные данные** — CRM, платёжная система, ERP (если предоставлены).
2. **Веб-поиск** — бенчмарки по отрасли (средний чек, частота покупок, retention rate).
3. **Отраслевые отчёты** — типичные RFM-распределения для данного типа бизнеса.
4. **Аналитика продукта** — данные активности пользователей (для SaaS — логины, действия).
5. **Данные от пользователя** — выгрузки из CRM, Google Analytics, платёжного провайдера.

> При работе с моделированными данными — указать источники бенчмарков.

## Протокол

### Шаг 0 — Сбор контекста
Проверить наличие обязательных входных данных. Определить, работаем с реальными данными или моделированными. Зафиксировать период анализа.

### Шаг 1 — Определение области данных
- Определить единицу анализа: клиент / аккаунт / подписчик.
- Определить «транзакцию»: покупка / оплата подписки / значимое действие.
- Определить период: начальная и конечная дата.
- Исключить аномалии: тестовые аккаунты, возвраты, аномальные суммы.

### Шаг 2 — Recency Score (R)
Оценить давность последней транзакции для каждого клиента:
- Рассчитать дни с последней транзакции.
- Разбить на квинтили (5 групп) или использовать бизнес-логику.
- R=5: недавняя активность (например, 0-30 дней).
- R=1: давно не было транзакций (например, >365 дней).
- Пороги зависят от бизнес-цикла (ежедневные покупки vs годовые контракты).

### Шаг 3 — Frequency Score (F)
Оценить частоту транзакций за период:
- Подсчитать количество транзакций на клиента.
- Разбить на квинтили (5 групп).
- F=5: очень частые покупки.
- F=1: однократная или крайне редкая покупка.
- Калибровка: для SaaS — ежемесячные оплаты; для e-commerce — заказы.

### Шаг 4 — Monetary Score (M)
Оценить суммарный объём транзакций за период:
- Рассчитать общую сумму на клиента.
- Разбить на квинтили (5 группы).
- M=5: топовые по сумме.
- M=1: минимальные траты.
- Решить: общая сумма или средний чек (зависит от бизнес-модели).

### Шаг 5 — Сегментация
Присвоить каждому клиенту RFM-код (три цифры от 111 до 555) и сгруппировать в именованные сегменты:
- **Champions** (R=5, F=5, M=5): лучшие клиенты.
- **Loyal Customers** (R=4-5, F=3-5, M=3-5): регулярные и ценные.
- **Potential Loyalists** (R=4-5, F=1-3, M=1-3): новые с потенциалом.
- **Recent Customers** (R=5, F=1, M=1): только пришли.
- **Promising** (R=3-4, F=1-2, M=1-2): есть интерес, нужна активация.
- **Need Attention** (R=3, F=3, M=3): средние, рискуют уйти.
- **About to Sleep** (R=2-3, F=1-2, M=1-2): теряют интерес.
- **At Risk** (R=1-2, F=3-5, M=3-5): были активны, уходят.
- **Cannot Lose** (R=1, F=5, M=5): VIP, которые ушли.
- **Hibernating** (R=1-2, F=1-2, M=1-2): давно неактивны.
- **Lost** (R=1, F=1, M=1): потеряны.

### Шаг 6 — Стратегия по сегментам
Для каждого сегмента определить:
- Цель (удержание, реактивация, upsell, onboarding).
- Ключевое действие (email, скидка, персональное предложение, звонок).
- Бюджетный приоритет (высокий / средний / низкий).
- KPI для оценки эффективности.

### Шаг 7 — Персонализация
Для каждого сегмента рекомендовать:
- Тон коммуникации.
- Тип предложения (скидка, новый функционал, эксклюзив).
- Канал коммуникации (email, push, SMS, звонок).
- Частоту контакта.

## Валидация (Quality Gate)

- [ ] Определены все три метрики (R, F, M) с чёткими порогами
- [ ] Пороги квинтилей обоснованы бизнес-циклом
- [ ] Выделены минимум 6 именованных сегментов
- [ ] Для каждого сегмента указана стратегия и конкретные действия
- [ ] Если данные моделированные — это явно указано
- [ ] Указаны источники бенчмарков при моделировании
- [ ] Рекомендации по персонализации привязаны к сегментам
- [ ] Нет логических противоречий в сегментации (RFM-коды корректны)

> Если валидация не пройдена — доработать до повторного прохождения, а не передавать дальше.

## Handoff
Результат `$rfm-analysis` является входом для:
- **Стратег / Медиатор** — сегменты клиентов для стратегических решений.
- `$unit-economics` — LTV по сегментам, retention rates.
- `$cohort-analysis` — группировка по когортам внутри сегментов.
- `$customer-journey-mapping` — точки воздействия на разных этапах для каждого сегмента.
- `$icp-buyer-persona` — обогащение персон поведенческими данными.

При передаче — использовать `$handoff` с таблицей сегментов и стратегиями.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Одинаковые пороги для всех бизнесов | У SaaS и e-commerce разные циклы покупки | Калибровать пороги под бизнес-модель |
| Моделированные данные без пометки | Решения на основе гипотез, а не фактов | Явно помечать гипотетические данные |
| Слишком много сегментов | Невозможно создать уникальную стратегию для каждого | 6-11 именованных сегментов максимум |
| Сегментация без стратегии | RFM ради RFM, нет actionable-выводов | Каждый сегмент = конкретное действие |
| Игнорирование M при подписке | В SaaS Monetary менее вариативна | Заменить M на engagement score для SaaS |
| Статичный анализ | RFM меняется каждый месяц | Рекомендовать регулярный пересчёт |
| Без персонализации | Одинаковое сообщение для Champions и At Risk | Разный тон, канал и предложение по сегментам |
