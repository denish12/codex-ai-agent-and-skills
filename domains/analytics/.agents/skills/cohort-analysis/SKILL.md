# Когортный анализ — Retention, revenue и паттерны поведения по группам

## Когда использовать
- При анализе удержания клиентов — понять, как retention меняется со временем.
- При оценке эффективности изменений — сравнить когорты до и после изменений.
- При прогнозировании LTV — когортные данные дают наиболее точную оценку.
- При выявлении проблем в онбординге — ранний churn в новых когортах.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Продукт / сервис | ✅ | Что анализируем |
| Б��знес-модель | ✅ | SaaS / E-commerce / Marketplace / App |
| Критерий когортирования | ✅ | По дате регистрации / первой покупки / другое |
| Данные по когортам | ⬚ | Реальные данные retention / revenue (если есть) |
| Период анализа | ⬚ | За какой период (по умолчанию — 12 месяцев) |
| Гранулярность | ⬚ | Недели / месяцы / кварталы (по умолчанию — месяцы) |
| Размер когорт | ⬚ | Количество пользователей в каждой когорте |

> Если реальных данных нет — скил формирует **методологический фреймворк** с бенчмарками по отрасли, который можно применить при появлении данных.

## Источники данных
1. **Аналитика продукта** — данные о регистрациях, активности, оплатах.
2. **Веб-поиск** — отраслевые бенчмарки retention (Mixpanel, Amplitude, Lenny Rachitsky).
3. **Отраслевые отчёты** — средние retention curves по вертикалям.
4. **CRM / Payment data** — данные от пользователя, если предоставлены.
5. **Публичные данные** — retention кейсы публичных компаний (S-1 filings, earnings).

> При работе с бенчмарками — указать источник и степень применимости.

## Протокол

### Шаг 0 — Сбор контекста
Проверить наличие обязательных данных. Определить, работаем с реальными данными или создаём методологический фреймворк. Зафиксировать период и гра��улярность.

### Шаг 1 — Определение критериев когортирования
- **Основной критерий:** дата регистрации / первой покупки / подписки.
- **Период когорты:** неделя / месяц / квартал.
- **Метрика активности:** что считаем «активным» (логин, транзакция, ключевое действие).
- **Период наблюдения:** на какую глубину смотрим (6 / 12 / 24 месяца).
- Исключения: тестовые аккаунты, боты, аномалии.

### Шаг 2 — Retention Table (Треугольник удержания)
Построить таблицу retention по когортам:
- Строки: когорты (по месяцам регистрации).
- Столбцы: месяцы жизни (Month 0, Month 1, ... Month N).
- Значения: процент активных пользователей от исходного размера когорты.
- Month 0 = 100% (по определению).

### Шаг 3 — Retention Curves
Визуализировать кривые удержания:
- Наложить кривые разных когорт для сравнения.
- Определить характерную форму:
  - **Flattening curve** — выход на плато (здоровый продукт).
  - **Continuous decay** — постоянное снижение (проблема retention).
  - **Smile curve** — снижение, затем рост (реактивация).
- Выявить точку стабилизации (если есть).

### Шаг 4 — Паттерны и аномалии
Проанализировать когорты на предмет паттернов:
- **Plateau (плато):** на каком месяце retention стабилизируется.
- **Decay (затухание):** какие когорты теряют пользователей быстрее.
- **Revival (возврат):** есть ли сезонные или триггерные возвраты.
- **Улучшение/ухудшение:** новые когорты лучше или хуже старых.
- Привязать аномалии к событиям (запуск фичи, смена цен, маркетинговая акция).

### Шаг 5 — Сравнение когорт
Сравнить когорты по ключевым срезам:
- По времени: ранние vs поздние (улучшается ли продукт).
- По каналу привлечения: если данные позволяют.
- По сегменту: разные тарифы, геозоны, устройства.
- Определить, какие факторы коррелируют с лучшим retention.

### Шаг 6 — Revenue per Cohort
Если доступны данные о выручке:
- Построить таблицу cumulative revenue per cohort.
- Рассчитать ARPU по когортам.
- Определить, когда когорта «окупает» CAC.
- Выявить когорты с наибольшей и наименьшей выручкой.

### Шаг 7 — Actionable Insights и Action Plan
На основе анализа сформировать:
- Ключевые findings (3-5 главных выводов).
- Корневые причины (гипотезы о том, почему retention такой).
- Конкретные действия для улучшения retention.
- Метрики для отслеживания эффекта.
- Рекомендации по частоте пересчёта.

## Валидация (Quality Gate)

- [ ] Определён критерий когортирования с обоснованием
- [ ] Retention table построена (или методологический фреймворк с бенчмарками)
- [ ] Retention curves описаны с определением формы (plateau/decay/smile)
- [ ] Выявлены паттерны и аномалии с привязкой к событиям
- [ ] Проведено сравнение когорт минимум по одному срезу
- [ ] Сформулированы actionable insights (минимум 3)
- [ ] Action plan содержит конкретные действия с метриками
- [ ] Если данные моделированные — это явно указано

> Если валидация не пройдена — доработать до повторного прохождения, а не передавать дальше.

## Handoff
Результат `$cohort-analysis` является входом для:
- **Стратег / Медиатор** — retention как фундамент для стратегии роста.
- `$unit-economics` — когортный LTV для точного расчёта.
- `$rfm-analysis` — пересечение когортных данных с RFM-сегментами.
- `$customer-journey-mapping` — точки оттока для оптимизации пути.
- `$trend-analysis` — тренды в retention как индикатор рыночных изменений.

При передаче — использов��ть `$handoff` с ключевыми findings и retention curves.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Средний retention без когорт | Маскирует проблемы новых когорт | Всегда разбивать на когорты |
| Когорты без контекста | Цифры без объяснения «почему» | Привязывать аномалии к событиям |
| Слишком мелкие когорты | Статистическая незначимость | Минимум 30-50 пользователей на когорту |
| Только Day 1 / Day 7 / Day 30 | Пропущены промежуточные паттерны | Полная retention table с каждым периодом |
| Без action plan | Анализ ради анализа | Каждый finding = конкретное действие |
| Игнорирование revenue per cohort | Retention без привязки к деньгам | Если данные есть — считать revenue per cohort |
| Статичный анализ | Когорты нужно пересчитывать | Рекомендовать ежемесячный пересчёт |
