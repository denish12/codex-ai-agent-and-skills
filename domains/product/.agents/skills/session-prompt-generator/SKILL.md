# Session Prompt Generator — Генерация промпта для следующей сессии

## Когда использовать
- При **завершении сессии** в Full Pipeline A (`/ship-right-thing`) или Full Pipeline B (`/shape-prioritize`) — перед межсессионным `$handoff`.
- При **необходимости передать проект** другому оператору — формирование самодостаточного промпта.
- При **запросе пользователя** «сгенерируй промпт для продолжения» — в любой момент.
- При **аварийном завершении** сессии — сохранение максимума контекста для восстановления.

> **Ключевой принцип:** следующая сессия начинается с **НУЛЯ** контекста. Промпт должен содержать всё необходимое, чтобы новая сессия продолжила работу без потерь.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Состояние `$board` | ✅ | Текущая доска задач со всеми статусами |
| Завершённые deliverables | ✅ | Полные артефакты текущей и предыдущих сессий |
| Текущий гейт | ✅ | Где остановились (ID гейта из `$board`) |
| Следующий гейт | ✅ | Куда переходим (ID следующего гейта) |
| Adversarial camp | ⬚ | Для Full A: Customer-Champion (α) / Business-Champion (β). Для Full B: Build-Camp (α) / Cut-Camp (β). Определяет фильтрацию artefacts |
| Открытые вопросы | ⬚ | Блокеры, неразрешённые конфликты, вопросы к пользователю |
| Номер текущей сессии | ✅ | Какая сессия завершается (1-6) |
| Название инициативы | ✅ | Название продуктовой инициативы |

## Протокол

### Шаг 0 — Сбор состояния
1. Прочитать текущее состояние `$board` — все гейты, статусы, deliverables.
2. Собрать все завершённые артефакты текущей сессии.
3. Загрузить кумулятивный индекс из предыдущих сессий (если есть).
4. Определить следующий гейт и его требования из `$gates`.
5. Для Full A/B — определить camp следующей сессии (α / β / Mediator-neutral).
6. По маппингу «Сессия → Гейты → Агенты → Скилы» определить, что загружать.

### Шаг 1 — Определение критических данных
1. Выделить данные, которые **невосстановимы** без предыдущего контекста:

   | Категория | Примеры | Приоритет |
   |-----------|---------|:---------:|
   | Ответы пользователя | Discovery interviews (DISC-01), уточнения, решения | 🔴 Critical |
   | JTBD canvas / problems | Канва, топ-3 формулировки проблем | 🔴 Critical |
   | Цитаты-доказательства | Дословные цитаты клиентов | 🔴 Critical |
   | Strategy Briefs (Full A) | Стратегия Alpha + Beta | 🔴 Critical |
   | Scope proposals (Full B) | Предложения Build + Cut | 🔴 Critical |
   | Mediator synthesis | Единая стратегия / scope + обоснование | 🔴 Critical |
   | PRD content | User stories + AC | 🔴 Critical |
   | Решения пользователя | Одобрено, отклонения, изменения scope | 🟡 Important |
   | Метрики прогресса | Время на гейты, блокеры | 🟢 Nice-to-have |

2. **Правило:** все 🔴 Critical данные включаются **полным текстом**. Без исключений.
3. **Adversarial filter (Full A/B):** если следующая сессия — противоположная camp (Beta после Alpha, ещё до Mediator), artefacts противоположной camp **ИСКЛЮЧАЮТСЯ** из промпта. Это прямое следствие camp independence rule.

### Шаг 2 — Определение агентов и скилов
1. По маппингу определить: какие агенты и скилы нужны в следующей сессии.
2. Сформировать список файлов для загрузки (конкретные пути).
3. Для фреймворк-скилов: указать конкретные скилы, если уже известны (из Strategy Brief или PRD).

### Шаг 3 — Формирование промпта
1. Заполнить шаблон промпта (6 блоков — см. ниже).
2. Включить ПОЛНЫЙ текст критических данных (не ссылки, не резюме).
3. Сформулировать конкретную задачу для следующей сессии.
4. Указать, какие агенты и скилы активировать (конкретные пути к файлам).
5. Для Full A/B — явно указать camp (Alpha / Beta / neutral).

### Шаг 4 — Валидация промпта
Тест самодостаточности — для каждого пункта ответить «можно ли без внешнего контекста?»:

| # | Проверка | Результат |
|---|----------|:---------:|
| 1 | Можно начать работу, имея ТОЛЬКО этот промпт? | Да / Нет |
| 2 | Нет ссылок на «предыдущий контекст» / «как обсуждалось»? | Да / Нет |
| 3 | Все 🔴 Critical данные включены полным текстом? | Да / Нет |
| 4 | Задача для сессии конкретна (гейт, действие, deliverable)? | Да / Нет |
| 5 | Агенты и скилы перечислены с путями к файлам? | Да / Нет |
| 6 | Доска задач включена? | Да / Нет |
| 7 | Открытые вопросы зафиксированы? | Да / Нет |
| 8 | Camp явно указан (Full A/B)? | Да / Нет |
| 9 | Артефакты противоположной camp исключены (Full A/B)? | Да / Нет |

**Если хотя бы один = Нет — доработать промпт.**

### Шаг 5 — Сохранение
1. Включить промпт в файл `$handoff`: `docs/product/session-N-handoff.md`.
2. Показать промпт пользователю для сохранения / копирования.
3. Предупредить: «Этот промпт — вставьте его в начале следующей сессии».

### Шаг 6 — Аварийный режим
Если сессия завершается внезапно (таймаут, ошибка, потеря контекста):

1. **Немедленно** сохранить всё, что доступно: доску, последний deliverable, открытые вопросы.
2. Сформировать **аварийный промпт** (минимальная версия):
   - Блок 1 (идентификация) — обязателен.
   - Блок 5 (состояние доски) — обязателен.
   - Блок 4 (данные) — всё, что удалось собрать.
   - Блок 6 (задача) — «Восстановить контекст из session-N-handoff.md, проверить целостность данных, продолжить с гейта [ID]».
3. Пометить промпт как **⚠️ АВАРИЙНЫЙ — возможна потеря данных**.
4. Указать, какие данные могли быть потеряны и как восстановить (повторный user interview, запрос у пользователя).

## Структура генерируемого промпта

### Блок 1 — Идентификация инициативы
- Название инициативы.
- Режим пайплайна (/ship-right-thing / /shape-prioritize).
- Номер следующей сессии.
- Какие сессии завершены.
- Camp следующей сессии (Alpha / Beta / Mediator-neutral / —).

### Блок 2 — Инструкция по загрузке
- Прочитать `domains/product/AGENTS.md`.
- Загрузить агентов, которые нужны в этой сессии (из маппинга).
- Загрузить скилы: `$board`, `$handoff`, `$gates` + специфичные для гейтов.
- Конкретные пути к файлам.

### Блок 3 — Выполненная работа (резюме)
- Что было сделано в предыдущих сессиях (2-3 предложения на сессию).
- Ключевые решения и их обоснования.

### Блок 4 — Критические данные
- ПОЛНЫЙ ТЕКСТ ключевых артефактов, необходимых для продолжения.
- Discovery Brief, JTBD canvas, Strategy Briefs, Mediator synthesis.
- Ответы пользователя на interview (если были).
- **Camp filter:** artefacts противоположной camp исключены (Full A/B, до Mediator).

### Блок 5 — Текущее состояние
- Доска задач (`$board`) в текущем состоянии.
- Какой гейт следующий.
- Открытые вопросы и блокеры.

### Блок 6 — Задача для сессии
- Конкретная формулировка: что нужно сделать в этой сессии.
- Ожидаемый результат (deliverables).
- Ограничения и особенности (camp lens для Full A/B).

## Правила формирования промпта

### Обязательные правила
1. **Самодостаточность** — промпт должен работать без ЛЮБОГО предыдущего контекста.
2. **Полнота данных** — включать ПОЛНЫЙ текст, не ссылки и не резюме.
3. **Конкретность задачи** — не «продолжи работу», а «выполни STRAT-02β: сформируй Business-Champion Strategy Brief на основе Discovery Brief, игнорируй Camp Alpha artefacts».
4. **Структурированность** — чёткие блоки, маркированные заголовками.
5. **Загрузка скилов** — явно указать какие агенты и скилы загрузить с путями к файлам.
6. **Adversarial independence** — для Full A/B при handoff к противоположной camp: явно зафиксировать исключение artefacts.

### Запрещённые элементы
1. Ссылки на «предыдущую сессию» без полного содержания.
2. «Как мы обсуждали» — нет контекста, нечего обсуждать.
3. «См. результаты выше» — нет «выше», всё должно быть в промпте.
4. Сокращённые данные — «основные выводы: ...». Нужны ВСЕ выводы.
5. Предположения о контексте — «вы уже знаете, что...».
6. В Full A/B: смешение artefacts разных camp'ов до Mediator.

```
   - **NSM** — метрика бизнес-ценности (пример: «ARR from accounts with AI summarization enabled на >60% teams»)
   - **OKR** — квартальные цели с акцентом на revenue / NRR / competitive moat
   - **Roadmap** — темы с обоснованием бизнес-ROI
   - **Гипотеза ценообразования** — конкретная модель (per-seat premium? usage-based?)

**Ожидаемый результат:**
`strategy-brief-beta.md` (~800-1200 слов) с явным аргументом через business-lens. Включить:
- Почему сейчас (рыночный тайминг)
- Модель влияния на выручку (сценарные прогнозы на 3 года)

- История NRR / expansion

**Ограничения:**
- **НЕ** притворяться Customer-Champion lens
- **НЕ** запрашивать артефакты Camp Alpha (они хранятся у Conductor до MED-01)
- Mediator (MED-01, Сессия 4) синтезирует — твоя задача представить наиболее острый бизнес-кейс

→ Начни как Conductor (COND-03). Проверь изоляцию camp, активируй Product Strategist (β). Передай Discovery Brief (ТОЛЬКО) как input.

```

## Валидация (Quality Gate)

Промпт считается корректным если:

- [ ] Промпт самодостаточен — можно начать работу без внешнего контекста
- [ ] Название инициативы и режим указаны
- [ ] Номер сессии и история сессий указаны
- [ ] Camp явно указан (Full A/B)
- [ ] Агенты и скилы для загрузки перечислены с путями к файлам (из маппинга)
- [ ] Все 🔴 Critical данные включены полным текстом
- [ ] Доска задач (`$board`) включена в текущем состоянии с метриками
- [ ] Следующий гейт и задача сформулированы конкретно (гейт, действие, deliverable)
- [ ] Camp filter применён: artefacts противоположной camp явно исключены (Full A/B, до Mediator)
- [ ] Открытые вопросы и блокеры зафиксированы
- [ ] Нет запрещённых элементов (ссылки без содержания, сокращения, предположения)
- [ ] Тест валидации (9 пунктов, шаг 4) — все «Да»
- [ ] При аварийном режиме — промпт помечен ⚠️, указаны потерянные данные

## Handoff
Результат `$session-prompt-generator` является входными данными для:
- **Следующая сессия** — через копирование промпта пользователем.
- **`$handoff`** — промпт включается в файл `docs/product/session-N-handoff.md`.
- **`$board`** — состояние доски сохраняется в промпте.

Формат передачи: промпт в markdown, готовый к вставке в начало новой сессии.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Промпт без Discovery данных | Следующая сессия теряет customer-centric foundation | Включить ПОЛНЫЙ текст Discovery Brief |
| «Продолжи работу» | Нет контекста, с чего продолжать | Конкретная задача: гейт, действие, deliverable |
| Ссылки на файлы без содержания | Файлы могут быть недоступны | Включить содержание прямо в промпт |
| Промпт без доски задач | Непонятно, где мы в пайплайне | Всегда включать `$board` с метриками |
| Сокращённые артефакты | Потеря критических деталей | Полный текст каждого 🔴 Critical артефакта |
| Пропуск открытых вопросов | Проблемы всплывут снова | Зафиксировать все блокеры и вопросы |
| Нет инструкции по загрузке | Сессия не загрузит нужных агентов | Маппинг → конкретные пути к файлам |
| Промпт > 50% «воды» | Размытие критических данных | Структура: блоки, таблицы, конкретика |
| Аварийное завершение без промпта | Полная потеря контекста | Аварийный режим: минимальный промпт + пометка ⚠️ |
| Camp contamination в промпте | Артефакты противоположной camp попадают в промпт → независимость нарушена | Явный фильтр + пометка «ИСКЛЮЧЕНО» для отсутствующих артефактов |
