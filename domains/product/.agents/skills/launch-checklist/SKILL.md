# Launch Checklist (B2B SaaS)

> **Категория:** Release  ·  **Slug:** `launch-checklist`

## Когда использовать

- За 2-4 недели до запуска feature / product / tier.
- Для phased rollout planning.
- Для alignment Product + Eng + CS + Sales + Marketing.
- Как quality gate перед GA.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Фича / инициатива | ✅ | Что запускаем |
| Целевая дата запуска | ✅ | Планируемая дата |
| Заинтересованные стороны | ✅ | PM, Eng, CS, Sales, Marketing, Legal |
| Уровень влияния на клиентов | ✅ | Minor / Major / Breaking |
| Регуляторные последствия | ⬚ | GDPR, отраслевые |

## Источники данных

1. `$prd-template` — scope фичи.
2. `$gtm-brief` — внешние сообщения.
3. `$release-notes` — анонс для клиентов.
4. `$rollback-plan` (если отдельный) — запасной план.

### Связь с другими скилами

| Скил | Что берём | Когда вызывать |
|------|-----------|----------------|
| `prd-template` | Что запускаем | Родительский вход |
| `gtm-brief` | Внешние сообщения | Часть запуска |
| `release-notes` | Для клиентов | Часть запуска |
| `ab-test-design` | Дизайн поэтапного rollout | Если A/B запуск |

## Launch Phases

1. **T-4 недели: Внутренняя подготовка**
2. **T-2 недели: Мягкий запуск (dogfood + бета-клиенты)**
3. **T-1 неделя: Укрепление перед запуском**
4. **T-0: День запуска**
5. **T+1 неделя: Мониторинг + итерация**
6. **T+4 недели: Ревью после запуска**

## Checklist

### 🔧 Internal Preparation (T-4 weeks)

**Engineering:**
- [ ] Feature flag реализован + протестирован
- [ ] Observability: дашборд метрик создан
- [ ] Алерты настроены для error rate, latency, usage
- [ ] Нагрузочное тестирование выполнено (для высокотрафиковых фич)
- [ ] Проверка безопасности завершена
- [ ] Rollback протестирован (откат работает end-to-end)

**Product:**
- [ ] PRD финализирован
- [ ] Метрики успеха определены + инструментированы
- [ ] Внутреннее демо готово
- [ ] Edge cases задокументированы

**Legal / Compliance:**
- [ ] Обновления ToS / Политики конфиденциальности (если нужно)
- [ ] Обновления DPA (если изменяется обработка данных)
- [ ] Соответствие отраслевым требованиям (SOC2, HIPAA, GDPR) проверено
- [ ] Вопросы экспортного контроля (если применимо)

**Data:**
- [ ] Аналитическое инструментирование активно
- [ ] Дашборд создан для мониторинга после запуска
- [ ] Базовые метрики зафиксированы (для сравнения до/после)

### 🧪 Soft Launch (T-2 weeks)

**Dogfooding:**
- [ ] Внутренняя команда использует фичу 5+ дней
- [ ] Обратная связь собрана от внутренних пользователей
- [ ] Топ-3 точки трения решены или задокументированы

**Бета-клиенты:**
- [ ] Бета-группа определена (5-20 клиентов, разные сегменты)
- [ ] Бета-соглашение подписано (если меняет ToS)
- [ ] Опрос обратной связи + интервью сессии
- [ ] Команда поддержки проинструктирована о возможных проблемах

**Итерация:**
- [ ] Сессия обзора бета-обратной связи
- [ ] Критические проблемы решены
- [ ] Feature gates скорректированы

### 📢 Pre-Launch Hardening (T-1 week)

**Коммуникации с клиентами:**
- [ ] Release notes финализированы (через `$release-notes`)
- [ ] Email-анонс подготовлен
- [ ] In-app баннер анонса разработан
- [ ] Блог-пост / changelog опубликован
- [ ] Документация (публичная база знаний) обновлена
- [ ] Видео-инструкция (если крупная фича)

**Sales Enablement:**
- [ ] Sales deck обновлён
- [ ] Battle cards (против конкурентов) обновлены
- [ ] Скрипт демо / окружение готово
- [ ] Обновление ценообразования (если применимо) согласовано

**CS / Support Enablement:**
- [ ] Сессия обучения проведена
- [ ] Внутренний FAQ / runbook создан
- [ ] Категории тикетов поддержки обновлены
- [ ] Путь эскалации для проблем запуска

**Marketing:**
- [ ] Landing page обновлён / новая страница готова
- [ ] Посты в социальных сетях запланированы
- [ ] PR outreach (если крупный запуск)
- [ ] Партнёрские анонсы (если применимо)

### 🚀 Launch Day (T-0)

**Утро запуска:**
- [ ] Финальная проверка работоспособности (дашборд метрик зелёный)
- [ ] Команда поддержки на дежурстве
- [ ] PM мониторит метрики (первые 2 часа интенсивно)
- [ ] Feature flag включён (% rollout по плану: 5% → 25% → 100%)

**Коммуникации:**
- [ ] Email отправлен списку клиентов
- [ ] In-app баннер активирован
- [ ] Блог-пост опубликован
- [ ] Обновление changelog активно
- [ ] Посты в социальных сетях опубликованы
- [ ] Команда Sales уведомлена

**Мониторинг:**
- [ ] Error rate в пределах базового уровня
- [ ] Latency в пределах SLA
- [ ] Adoption использования отслеживается
- [ ] Объём тикетов поддержки мониторится

### 📊 Post-Launch Monitoring (T+1 week)

**Ежедневно:**
- [ ] Обзор дашборда метрик
- [ ] Обзор трендов тикетов поддержки
- [ ] Проверка NPS обратной связи

**Конец первой недели:**
- [ ] Проверка метрики успеха (базовый уровень → текущий)
- [ ] Adoption rate
- [ ] Обнаруженные проблемы + залогированы
- [ ] Темы обратной связи клиентов

### 🔄 Post-Launch Review (T+4 weeks)

- [ ] Метрики успеха vs цели
- [ ] Анализ adoption по сегменту
- [ ] Темы тикетов поддержки
- [ ] Влияние на churn / расширение
- [ ] Документ извлечённых уроков
- [ ] Корректировки roadmap на основе выводов

## План Rollback (Всегда обязателен)

- [ ] Условия триггера определены:
  - Error rate > X% в течение Y минут
  - Latency > SLA в течение Y минут
  - Критическая эскалация от клиента
  - Инцидент безопасности
- [ ] Механизм rollback: отключение feature flag (предпочтительно) или git revert
- [ ] Время выполнения rollback: цель ≤ 5 минут
- [ ] План коммуникации: кто уведомляет клиентов, через какой канал
- [ ] После rollback: анализ первопричины, план повторной попытки

## Валидация (Quality Gate)

- [ ] Все секции чеклиста заполнены
- [ ] Feature flag + rollback протестированы
- [ ] Observability инструментирована
- [ ] Бета-обратная связь учтена
- [ ] Коммуникации готовы (email, блог, документы, соцсети)
- [ ] Sales + CS подготовлены
- [ ] План rollback с условиями триггера
- [ ] Ревью после запуска запланировано

## Handoff

Результат является входом для:
- **Release Gate** (Conductor) — подписание
- **Engineering** → деплой
- **CS / Sales / Marketing** → внешнее исполнение
- **Data Analyst** → мониторинг после запуска

Формат: launch checklist markdown + tracking board. Через `$handoff`.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Big-bang запуск | Высокий риск, нет сети безопасности | Поэтапный rollout (5% → 100%) |
| Нет плана rollback | Перебой затягивается | Всегда иметь Plan B |
| Нет беты | Проблемы обнаруживаются в продакшене | Бета 2+ недели |
| Нет обучения CS | Тикеты захлёстывают поддержку | Внутреннее обучение T-1 неделю |
| Нет observability | Слепота после запуска | Метрики + алерты до запуска |
| Игнорировать legal | Нарушения соответствия | Legal review для любых изменений данных/контракта |
| Один человек владеет всем | Единая точка отказа | Чёткий владелец по каждой секции |

## План Rollback

### Условия триггера (ЛЮБОЕ из):
- **Error rate > 2% устойчиво в течение > 15 минут** → немедленное отключение флага
- **P95 latency > 120с устойчиво в течение > 30 минут** → расследование + рассмотреть отключение
- **Критическая эскалация от Enterprise аккаунта** → проверка в течение 2 часов
- **Инцидент безопасности (утечка данных / несанкционированный доступ)** → немедленное отключение + уведомление клиентов
- **Нарушение SLA LLM провайдера (и основной, и резервный)** → автоматический failover + расследование

### Механизм Rollback
1. **Основной:** Отключение LaunchDarkly feature flag («ai_summarization_ga» → false)
2. **Время rollback:** ≤ 5 минут (распространение флага + инвалидация кэша)
3. **Влияние на пользователей:** AI переключатель серый, существующие AI summary остаются доступными (только чтение)
4. **Влияние на данные:** Нет потери данных — summary сохранены, только генерация приостановлена

### План коммуникации (при rollback)
- **Внутренний:** Slack #incident-response + подтверждение PagerDuty
- **Клиенты:** In-app баннер «AI Summarization временно приостановлено — расследуем» (в течение 10 мин)
- **Уведомление затронутых клиентов:** Email Enterprise клиентам в течение 1 часа
- **Страница статуса:** Публичный status.teamflow.com обновлён

### После Rollback
- Анализ первопричины в течение 48 часов (анализ 5 почему)
- Исправление проверено в staging + бета-клиентах
- План повторного запуска с выводами (может быть следующий день, следующая неделя или задержка)

> **Урок launch-checklist:** Checklist длиной в 100+ items — нормально для major B2B SaaS launch. **Rollback plan с numeric triggers** — «error rate >2% for >15 min» — не «if it breaks». **Ownership matrix** prevents «no one noticed» — каждый section has primary + backup. **Phased feature flag rollout** (Tier customers first, then upsell to Core) minimizes blast radius. **Communications в multi-channel** (email + in-app + blog + social + PR + sales enablement) — single channel never reaches all stakeholders.
