# Правила контент-пайплайна — Абсолютные ограничения

## Цель
Этот документ определяет непреложные правила работы контент-пайплайна. Нарушение любого из этих правил недопустимо. Правила применяются ко **всем** режимам пайплайна.

---

## Режимы пайплайна

### Decision Tree — выбор режима

```
Задача пользователя
       │
       ├── Новый контент с нуля / кампания / требует исследования?
       │   └── ДА → /start-content (Full Pipeline)
       │
       ├── Есть готовый текст, нужны правки?
       │   └── ДА → Правки > 50%?
       │           ├── ДА → /start-content (слишком много правок)
       │           └── НЕТ → /edit-content (Edit Pipeline)
       │
       ├── Простой пост, тема ясна, нет сложных фактов?
       │   └── ДА → Blast radius ≈ 0?
       │           ├── ДА → /quick-post (Quick Pipeline)
       │           └── НЕТ → /start-content или /edit-content
       │
       └── Не уверен?
           └── Выбрать более тяжёлый режим. Лучше перестраховаться.
```

### Дополнительные критерии

| Критерий | /start-content | /edit-content | /quick-post |
|----------|:-:|:-:|:-:|
| Нужно исследование данных | ✅ | ❌ | ❌ |
| Нужна стратегия (аудитория, платформа) | ✅ | ❌ | ❌ |
| Есть готовый текст | ❌ | ✅ | ❌ |
| Тема ясна, аудитория известна | — | ✅ | ✅ |
| Blast radius ≈ 0 | — | — | ✅ |
| Реакция на тренд / инфоповод | — | — | ✅ |
| Сложные фактические утверждения | ✅ | ❌ | ❌ |
| Целевое время | 1-3 часа | 30-60 мин | 5-15 мин |

### Схемы гейтов по режимам

**Full Pipeline** (`/start-content`):
```
CONDUCTOR → STRATEGIST → RESEARCHER → COPYWRITER → VISUAL_CONCEPT → REVIEWER → RELEASE GATE
   G1            G2            G3           G4            G5             G6          G7
```

**Edit Pipeline** (`/edit-content`):
```
CONDUCTOR → COPYWRITER → REVIEWER
   G1            G2          G3
```

**Quick Pipeline** (`/quick-post`):
```
CONDUCTOR → COPYWRITER + REVIEWER
   G1            G2          G3
```

## Гейты, deliverables и обязательные скилы

### Full Pipeline — `/start-content`

| Гейт | Агент | Deliverable | Обязательные скилы | Опциональные скилы |
|------|-------|-------------|-------------------|-------------------|
| G1 | **Conductor** | Handoff Envelope + доска | `$board`, `$handoff` | — |
| G2 | **Strategist** | Контент-бриф + контент-план | `$audience-analysis`, `$platform-strategy`, `$content-calendar`, `$content-brief` | `$competitor-content-analysis`, `$tone-of-voice` |
| G3 | **Researcher** | Пакет данных: факты, источники, нарратив | `$topic-research`, `$source-verification` | `$trend-research`, `$data-storytelling` |
| G4 | **Copywriter** | Финальный текст + заголовки + CTA | `$tone-of-voice`, `$headline-formulas`, `$cta-optimization` | `$storytelling-framework`, `$social-media-formats`, `$email-copywriting`, `$seo-copywriting` |
| G5 | **Visual Concept** | Визуальный бриф + AI-промпты | `$visual-brief`, `$platform-visual-specs` | `$moodboard`, `$image-prompt-engineering`, `$brand-guidelines` |
| G6 | **Reviewer** | APPROVED или список правок | `$content-review-checklist`, `$brand-compliance` | `$fact-checking`, `$readability-scoring`, `$platform-compliance` |
| G7 | **Release Gate** | Пакет для публикации | `$content-release-gate` | — |

### Edit Pipeline — `/edit-content`

| Гейт | Агент | Deliverable | Обязательные скилы |
|------|-------|-------------|-------------------|
| G1 | **Conductor** | Handoff + задача на правки | `$board`, `$handoff` |
| G2 | **Copywriter** | Отредактированный текст + diff | `$tone-of-voice` |
| G3 | **Reviewer** | APPROVED или доработки | `$content-review-checklist`, `$brand-compliance` |

### Quick Pipeline — `/quick-post`

| Гейт | Агент | Deliverable | Обязательные скилы |
|------|-------|-------------|-------------------|
| G1 | **Conductor** | Краткий бриф | `$board` |
| G2 | **Copywriter** | Готовый текст поста | `$tone-of-voice`, `$social-media-formats`, `$cta-optimization` |
| G3 | **Reviewer** | APPROVED или 1-2 правки | `$brand-compliance` |

## Правила (абсолютные)

### Правило 1: Явный Approved перед каждым переходом
Никогда не переходить к следующему гейту без явного **"Approved"** от пользователя.
- Даже если deliverable кажется очевидным.
- Даже при дедлайне.
- **Исключение:** `/quick-post` Gate 2 → Gate 3 может быть без паузы при явном запросе пользователя.

**Протокол Approved:**
1. Показать deliverable гейта пользователю.
2. Пройти `$gates` — проверить полноту и качество.
3. Запросить явное "Approved".
4. Только после Approved — выполнить `$handoff` → обновить `$board`.

### Правило 2: Нельзя пропускать гейты
Гейты нельзя пропускать, объединять или менять порядок без явного разрешения пользователя.
- Каждый гейт имеет свою ответственность и deliverable.
- Если гейт кажется избыточным — **обсудить** с пользователем, не пропускать молча.
- Порядок гейтов определён схемой режима. Нарушение порядка = нарушение правила.

### Правило 3: Каждый гейт должен иметь deliverable
Гейт не считается завершённым без конкретного deliverable (см. таблицы выше).
- Deliverable проверяется через `$gates` (Quality Gate чеклист).
- Гейт с пустым или неполным deliverable → `$gates` FAIL → возврат на доработку.
- Deliverable фиксируется в `$board` при закрытии гейта.

### Правило 4: Факты обязательно верифицируются
Любое фактическое утверждение в тексте должно быть верифицировано.
- Статистика → первичный источник обязателен (`$source-verification`).
- Цитаты → оригинальный контекст проверен (`$fact-checking`).
- Актуальность данных — **адаптивная** (не хардкод):
  - Tech / Digital: ≤ 1 год
  - Бизнес / Рынки: ≤ 2 года
  - Наука: ≤ 5 лет (для основ), ≤ 2 года (для прорывов)
  - Юридика: действующее законодательство (проверить дату)
- Нет источника → не использовать факт.
- **Исключение для `/quick-post`:** если в тексте нет фактических утверждений (мнение, юмор), `$fact-checking` не обязателен. Если есть хотя бы один факт — обязателен.

### Правило 5: Один пайплайн — одна задача
Каждый запуск пайплайна решает одну конкретную задачу.
- Не совмещать создание разных единиц контента в одном пайплайне.
- Если задача расширилась → завершить текущий пайплайн, запустить новый.
- Одна единица контента = один ID из `$content-calendar` (если применимо).

### Правило 6: Правки возвращаются к исполнителю
Если Reviewer дал правки → они возвращаются к **конкретному агенту** через `$handoff` (reverse):
- Текстовые правки → Copywriter.
- Визуальные правки → Visual Concept.
- Фактические ошибки → Researcher.
- Юридические вопросы → эскалация пользователю.
- **Conductor не переписывает текст.**
- Повторное ревью обязательно после правок (`$gates` → `$handoff` → Reviewer).

### Правило 7: Brand compliance — обязательная проверка
Каждая единица контента должна пройти `$brand-compliance`.
- Нет исключений, даже для `/quick-post`.
- Нарушение бренд-гайдлайнов = Blocker.
- Если `$brand-guidelines` не определены → запросить у пользователя **перед стартом** пайплайна.

### Правило 8: Платформа определяет формат
Контент адаптируется под платформу, не наоборот.
- Технические спеки проверяются через `$platform-visual-specs` и `$platform-compliance`.
- Нельзя публиковать контент в неправильном формате, размере или ratio.
- Текст адаптируется через `$social-media-formats`.

### Правило 9: Выбор режима — ответственность Conductor
Conductor определяет режим по Decision Tree (см. выше).
- Решение принимается на Gate 1 и **сообщается пользователю** с обоснованием.
- При сомнении — **выбирать более тяжёлый режим**.
- Пользователь может переопределить режим — Conductor подчиняется.
- Если в процессе выяснилось, что режим слишком лёгкий → предложить пользователю переключиться на более тяжёлый.

### Правило 10: CONTEXT.md обновляется после каждого пайплайна
После завершения пайплайна Conductor обновляет `domains/content/CONTEXT.md`:
- Режим пайплайна.
- Завершённые гейты.
- Ключевой результат (что было создано).
- Дата.

### Правило 11: `$board` обновляется при каждом переходе
Доска (`$board`) — единственный источник правды о текущем статусе:
- Создаётся Conductor'ом на Gate 1.
- Обновляется при **каждом** `$handoff` (forward и reverse).
- Статус гейта меняется только через `$board`.
- При завершении — все гейты `[✓]`.

### Правило 12: `$handoff` — единственный способ передачи
Передача между гейтами происходит **только** через `$handoff`:
- Forward handoff — после `$gates` PASS + user Approved.
- Reverse handoff — при `$gates` FAIL или Reviewer REQUIRES CHANGES.
- Каждый handoff фиксируется в лог передач.
- Handoff без `$gates` — запрещён.

## Параллелизм

### Что можно параллельно

| Гейты | Условие | Режим |
|-------|---------|-------|
| **Copywriter + Visual Concept** | Если текст не влияет на визуал (визуал по брифу, не по тексту) | Full Pipeline — с разрешения Conductor'а |
| **Copywriter + Reviewer** | Quick Pipeline: reviewer начинает сразу после copywriter | Quick Pipeline |

### Что нельзя параллельно

| Гейты | Почему |
|-------|--------|
| Strategist + Researcher | Researcher зависит от брифа Strategist'а |
| Copywriter + Reviewer (Full) | Reviewer проверяет текст Copywriter'а |
| Любые гейты без Approved | Правило 1 запрещает переход без Approved |

### Правило параллелизма
- Параллельная работа допустима **только** если выходы гейтов **не зависят** друг от друга.
- Conductor принимает решение о параллелизме и **сообщает** пользователю.
- Оба параллельных гейта требуют **отдельных** Approved.

## Эскалация

| Ситуация | Действие | Кто |
|----------|----------|-----|
| Блокер на гейте | Уведомить пользователя через `$board` (статус `[!]`), не переходить дальше | Текущий агент |
| Недостаточно данных | Reverse `$handoff` → Researcher с конкретными gap'ами | Copywriter / Reviewer |
| Бренд-гайдлайны не определены | Запросить у пользователя **перед стартом**, не начинать без них | Conductor |
| Источник не верифицирован | Удалить утверждение или найти замену через `$source-verification` | Researcher |
| Правки > 50% текста | Рекомендовать перезапуск `/start-content`, решение за пользователем | Reviewer / Conductor |
| Пользователь не отвечает | Ждать. Не переходить к следующему гейту без Approved. Напомнить через сообщение | Conductor |
| Дедлайн давит | Предложить переключиться на `/quick-post` или `/edit-content`. Не пропускать гейты | Conductor |
| Конфликт Reviewer ↔ Copywriter | Эскалировать пользователю с аргументами обеих сторон | Conductor |
| Режим оказался слишком лёгким | Предложить переключение на более тяжёлый. Решение за пользователем | Conductor |
| Неизвестная ситуация | Остановиться. Спросить пользователя. Не импровизировать | Conductor |

## Интеграция со скилами

### Оркестрационные скилы (используются на каждом гейте)

| Скил | Когда | Кто использует |
|------|-------|---------------|
| `$board` | Создание (G1), обновление (каждый переход), закрытие (финал) | Conductor + каждый агент при handoff |
| `$gates` | Перед каждым переходом — проверка deliverable | Conductor |
| `$handoff` | При каждом переходе (forward и reverse) | Conductor + текущий агент |

### Скилы по гейтам

Маппинг обязательных и опциональных скилов — см. таблицы в секции «Гейты, deliverables и обязательные скилы».

### Правило применения скилов
- **Обязательные** скилы гейта — выполняются всегда. Пропуск = `$gates` FAIL.
- **Опциональные** скилы — применяются по контексту задачи. Conductor или агент решает.
- Скил считается выполненным, если его **Quality Gate** (валидация) пройден.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Пропустить гейт «потому что просто» | Ошибки проходят незамеченными | Никогда не пропускать. Правило 2 |
| Переход без Approved | Пользователь не в курсе, потеря контроля | Всегда ждать Approved. Правило 1 |
| Conductor переписывает текст | Нарушение ролей, нет качественного ревью | Правки → Copywriter → Reviewer. Правило 6 |
| `/quick-post` для сложной темы | Нет исследования → ошибки в фактах | Decision Tree. При сомнении → тяжелее. Правило 9 |
| Handoff без `$gates` | Непроверенные deliverables → проблемы дальше | `$gates` PASS → `$handoff`. Правило 12 |
| Не обновлять `$board` | Потеря трекинга, непонятно где мы | Обновлять при каждом handoff. Правило 11 |
| Brand compliance пропущен | Контент не соответствует бренду → репутационный риск | Обязателен для всех режимов. Правило 7 |
| Параллельная работа зависимых гейтов | Результат одного гейта нужен другому → рассинхрон | Только независимые гейты параллельно |
| Импровизация в неизвестной ситуации | Непредсказуемые последствия | Остановиться, спросить пользователя |
