# Воркфлоу: /edit-content — Пайплайн редактирования контента

## Описание
Пайплайн для редактирования существующего контента. Используется когда основа уже есть, но требует улучшения: тон, структура, CTA, соответствие бренду, адаптация под платформу, визуальные правки.

## Когда использовать
- Есть готовый черновик текста.
- Нужно улучшить тон, структуру или CTA.
- Контент не прошёл ревью (возврат из `/start-content`).
- Требуется адаптация под новую платформу или аудиторию.
- Нужны визуальные правки (визуал уже есть, но не соответствует).

## Когда НЕ использовать
- Текст нужно переписать > 50% → `/start-content`.
- Нужно исследование данных или новая стратегия → `/start-content`.
- Простой пост с нуля, тема ясна → `/quick-post`.

## Вход воркфлоу

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Исходный текст | ✅ | Текущая версия контента (v1) |
| Задача на правки | ✅ | Что именно нужно изменить и почему |
| Платформа | ✅ | Где будет опубликовано (может отличаться от исходной) |
| Бренд-гайдлайны | ✅ | Из `$brand-guidelines` — для brand compliance |
| Персона ЦА | ⬚ | Из `$audience-analysis` — если адаптация под другую аудиторию |
| Исходный визуал | ⬚ | Если нужны визуальные правки |
| Предыдущий review | ⬚ | Если контент вернулся из ревью — список замечаний |

## Типы правок

| Тип | Глубина | Примеры | Гейты |
|-----|---------|---------|-------|
| **Тональные** | Лёгкая | Сменить ToV (формальнее / неформальнее), убрать клише | Conductor → Copywriter → Reviewer |
| **Структурные** | Средняя | Переструктурировать текст, улучшить hook, усилить CTA | Conductor → Copywriter → Reviewer |
| **Платформенные** | Средняя | Адаптировать с LinkedIn на Instagram, поменять формат | Conductor → Copywriter → Reviewer |
| **Визуальные** | Средняя | Поправить визуал, изменить композицию, обновить бренд-элементы | Conductor → Visual Concept → Reviewer |
| **Комплексные** | Высокая | Текст + визуал + платформа одновременно | Conductor → Copywriter → Visual Concept → Reviewer |
| **Post-review** | Зависит от замечаний | Исправления по замечаниям Reviewer'а | Conductor → Copywriter/Visual → Reviewer |

> Conductor определяет тип правок на Gate 1 и выбирает нужные гейты.

## Пайплайн

### Базовый (текстовые правки)
```
CONDUCTOR → COPYWRITER → REVIEWER → RELEASE CHECK
   G1            G2          G3           G4
```

### С визуальными правками
```
CONDUCTOR → VISUAL CONCEPT → REVIEWER → RELEASE CHECK
   G1             G2v            G3          G4
```

### Комплексный (текст + визуал)
```
CONDUCTOR → COPYWRITER → VISUAL CONCEPT → REVIEWER → RELEASE CHECK
   G1            G2            G2v             G3          G4
```

> Conductor на Gate 1 определяет какой вариант пайплайна активировать.

## Гейты и deliverables

### Gate 1: Conductor
**Вход:** Задача пользователя + исходный контент
**Действия:**
- Определить **тип правок** (тональные / структурные / платформенные / визуальные / комплексные / post-review)
- Зафиксировать текущую версию как **v1** (текст и/или визуал)
- Определить **какие гейты** нужны (базовый / визуальный / комплексный)
- Инициализировать доску (`$board`) для Edit Pipeline
- Сформулировать задачу: **что** менять, **почему**, **что НЕ трогать**
- Проверить: правки ≤ 50%? Если нет → предложить `/start-content`
**Обязательные скилы:** `$board`, `$handoff`
**Deliverable:** Handoff Envelope → Copywriter / Visual Concept
**Проверка:** `$gates` — задача конкретна, тип определён, v1 зафиксирована
**→ Ждать "Approved"**

### Gate 2: Copywriter (текстовые правки)
**Вход:** `$handoff` от Conductor с исходным текстом и задачей
**Действия:**
- Проверить/скорректировать ToV (`$tone-of-voice`)
- Улучшить заголовки (`$headline-formulas`) если нужно
- Переформатировать под платформу (`$social-media-formats`) если нужно
- Оптимизировать CTA (`$cta-optimization`) если нужно
- Внести запрошенные правки **с сохранением того, что не нужно менять**
- Подготовить **diff**: что именно изменено и почему
**Обязательные скилы:** `$tone-of-voice`
**Опциональные:** `$headline-formulas`, `$social-media-formats`, `$cta-optimization`, `$seo-copywriting`
**Deliverable:** Отредактированный текст (v2) + diff (список изменений с обоснованием)
**Проверка:** `$gates` — все запрошенные правки внесены, текст целостный, diff предоставлен
**→ Ждать "Approved"**

### Gate 2v: Visual Concept (визуальные правки)
**Вход:** `$handoff` от Conductor с исходным визуалом и задачей
**Действия:**
- Определить что менять в визуале по задаче
- Обновить `$visual-brief` с учётом правок
- Обновить AI-промпты (`$image-prompt-engineering`) если AI-генерация
- Проверить бренд-элементы (`$brand-guidelines`)
- Проверить спеки платформы (`$platform-visual-specs`)
**Обязательные скилы:** `$visual-brief`, `$platform-visual-specs`
**Опциональные:** `$image-prompt-engineering`, `$brand-guidelines`
**Deliverable:** Обновлённый визуальный бриф / промпты (v2) + описание изменений
**Проверка:** `$gates`
**→ Ждать "Approved"**

### Gate 3: Reviewer
**Вход:** `$handoff` от Copywriter / Visual Concept с v2
**Действия:**
- Чеклист ревью (`$content-review-checklist`) — **фокус на изменённых частях** + проверка что неизменённое не сломалось
- Бренд-compliance (`$brand-compliance`)
- Читаемость (`$readability-scoring`) — если текстовые правки
- Платформа-compliance (`$platform-compliance`)
- Сверка с исходной задачей: все запрошенные правки действительно внесены?
**Обязательные скилы:** `$content-review-checklist`, `$brand-compliance`
**Опциональные:** `$readability-scoring`, `$platform-compliance`, `$fact-checking`
**Deliverable:** APPROVED или список конкретных доработок с severity

**При APPROVED:** → Gate 4 (Release Check)
**При REQUIRES CHANGES:** → Reverse `$handoff` → Copywriter / Visual Concept → повторный Gate 3

**Проверка:** `$gates` — нет Blockers
**→ Ждать "Approved"**

### Gate 4: Release Check
**Вход:** APPROVED от Reviewer
**Действия:**
- Быстрая финальная проверка по `$content-release-gate` (облегчённая для Edit Pipeline):
  - Текст v2 финализирован ✅
  - Визуал v2 финализирован (если были визуальные правки) ✅
  - Brand compliance пройден ✅
  - Platform compliance пройден ✅
  - Пакет публикации обновлён ✅
- Обновить `$board` — все гейты `[✓]`
- Обновить `domains/content/CONTEXT.md`
**Deliverable:** Обновлённый пакет для публикации
**→ Финальное "Approved" → Готов к публикации**

## Reverse Handoff — протокол возврата

Если Reviewer дал **REQUIRES CHANGES**:

```
Reviewer (REQUIRES CHANGES)
    │
    ├── Текстовые правки → Reverse $handoff → Copywriter
    │   └── Copywriter вносит правки → v3 → Forward $handoff → Reviewer (повторный G3)
    │
    ├── Визуальные правки → Reverse $handoff → Visual Concept
    │   └── Visual Concept вносит правки → v3 → Forward $handoff → Reviewer (повторный G3)
    │
    └── Оба → Reverse $handoff → Copywriter + Visual Concept → Reviewer
```

**Правила reverse handoff:**
1. Указать **конкретные замечания** с severity (из `$content-review-checklist`).
2. Указать **что НЕ трогать** (уже одобренные части).
3. Обновить `$board` — гейт Reviewer → `[↩] Возврат`, целевой гейт → `[→] В работе`.
4. После исправлений — **повторная** `$gates` проверка → forward `$handoff` → Reviewer.
5. Максимум **3 итерации**. Если после 3 итераций не APPROVED → эскалация пользователю.

## Порог переключения на `/start-content`

| Индикатор | Как определить | Действие |
|-----------|---------------|----------|
| Правки > 50% текста | Diff показывает, что больше половины текста переписано | Предложить `/start-content` |
| Нужно новое исследование | В тексте нет данных, а задача требует фактов | Предложить `/start-content` с Researcher |
| Стратегия неверна | Не та аудитория, не та платформа, не тот CTA-фокус | Предложить `/start-content` со Strategist |
| 3+ итерации ревью | Copywriter не может довести до APPROVED | Эскалация → возможно, проблема в брифе → `/start-content` |

> Решение о переключении — за пользователем. Conductor **предлагает**, не переключает сам.

## Ограничения режима
- ❌ Не запускать исследование (`$trend-research`, `$topic-research`) — факты берём из исходника.
- ❌ Не пересматривать стратегию (аудитория, платформа) — только если пользователь явно попросил.
- ❌ Не создавать мудборд с нуля — только корректировать существующий визуал.
- ✅ Если в исправленном тексте появились **новые** фактические утверждения → `$fact-checking` обязателен.

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Переписать всё вместо точечных правок | Это `/start-content`, а не `/edit-content` | Сохранять то, что работает. Менять только запрошенное |
| Правки без diff | Непонятно что изменилось, Reviewer не может фокусированно проверить | Всегда предоставлять diff: что, где, почему |
| Не зафиксировать v1 | Нет возможности откатиться | Conductor фиксирует v1 на Gate 1 |
| Бесконечные итерации ревью | Время тратится, результат не улучшается | Максимум 3 итерации → эскалация |
| Визуальные правки без Visual Concept | Copywriter правит визуал = не его компетенция | Визуал → Visual Concept (Gate 2v) |
| Пропустить Release Check | «Ну это же просто правки» — но ошибки всё равно возможны | Gate 4 обязателен, даже облегчённый |
| Добавить новые факты без `$fact-checking` | Новые утверждения не верифицированы | Новые факты → обязательный `$fact-checking` |
| Не указать «что НЕ трогать» | Copywriter может сломать то, что уже одобрено | Conductor явно фиксирует scope правок |
