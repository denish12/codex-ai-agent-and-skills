# Handoff — Передача между гейтами

## Когда использовать
- При **каждом переходе** между гейтами — прямая передача (forward handoff).
- При **возврате на доработку** — обратная передача (reverse handoff).
- **Обязателен** на каждом переходе, в любом режиме пайплайна. Без исключений.

> **Разграничение:** `$gates` проверяет готовность. `$handoff` передаёт работу. `$board` отслеживает статус. Порядок: `$gates` PASS → `$handoff` → обновить `$board`.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Результат `$gates` | ✅ | PASS или CONDITIONAL PASS — handoff невозможен при FAIL |
| Агент-отправитель | ✅ | Кто передаёт (текущий гейт) |
| Агент-получатель | ✅ | Кому передаёт (следующий гейт) |
| Артефакты гейта | ✅ | Deliverables текущего гейта (из реестра `$gates`) |
| Режим пайплайна | ✅ | Full / Bugfix / Hotfix |
| Тип handoff | ✅ | Forward (прямая) / Reverse (возврат на доработку) |

> **Предусловие:** Handoff возможен **только** после `$gates` PASS или CONDITIONAL PASS. При FAIL — вернуть агенту, handoff не выполняется.

## Статусы передачи

| Статус | Значение | Когда |
|--------|----------|-------|
| **Завершён** | Все deliverables готовы, `$gates` PASS | Нормальный переход |
| **Условный** | Deliverables готовы с gap'ами, `$gates` CONDITIONAL PASS | Переход с пометкой: gap'ы зафиксированы |
| **Возврат** | Работа возвращается на доработку | `$gates` FAIL или feedback от следующего гейта / пользователя |
| **Блокер** | Передача невозможна, требуется эскалация | Внешний блокер (юридика, доступ, данные) |

## Реестр передач — что передавать

### Full Pipeline

| Переход | Артефакты | Ключевой контекст |
|---------|-----------|-------------------|
| **Conductor → Strategist** | Задача, scope, режим, доска | Бизнес-цель, бренд, ограничения |
| **Strategist → Researcher** | Audience analysis, platform strategy, content calendar, competitor analysis | Персоны, платформы, gaps конкурентов, приоритетные темы |
| **Researcher → Copywriter** | Topic research, source verification, data storytelling, content brief | Факты, источники, нарратив, бриф с тезисами |
| **Copywriter → Visual Concept** | Финальный текст, content brief (визуальная секция) | Тема, настроение, платформа, бренд-стиль |
| **Copywriter + Visual → Reviewer** | Финальный текст, визуал / visual brief, AI-промпты | Бриф для сверки, бренд-гайдлайны |
| **Reviewer → Release Gate** | Review checklist (APPROVED), brand compliance, fact checking, readability | Замечания (если были и исправлены), итоговые статусы |
| **Release Gate → Публикация** | Полный пакет: текст + визуал + мета + расписание | Дата, платформа, кросс-постинг |

### Bugfix Pipeline

| Переход | Артефакты | Ключевой контекст |
|---------|-----------|-------------------|
| **Conductor → Copywriter/Researcher** | Описание проблемы, scope исправления | Что сломано, где, приоритет |
| **Copywriter/Researcher → Reviewer** | Исправленный контент | Что именно изменилось, diff |
| **Reviewer → Release Gate** | Review checklist (APPROVED) | Фокус на исправленном |

### Hotfix Pipeline

| Переход | Артефакты | Ключевой контекст |
|---------|-----------|-------------------|
| **Conductor → Copywriter+Reviewer** | Описание проблемы, исправление | Blast radius ≈ 0 подтверждён, mini-checklist |

## Обратный Handoff (Reverse)

При возврате на доработку — специальный протокол:

### Когда происходит
- `$gates` — FAIL.
- Reviewer — REQUIRES CHANGES.
- Пользователь — запросил доработку.
- Следующий гейт — обнаружил проблему с артефактом предыдущего.

### Протокол reverse handoff
1. Указать **причину возврата** — конкретные gap'ы / замечания.
2. Указать **что именно доработать** — не «переделай», а «исправь CTA: вместо двух оставь один, формулировка по `$cta-optimization`».
3. Указать **что НЕ трогать** — что уже одобрено и не нуждается в изменениях.
4. Обновить `$board` — гейт-получатель → `[↩] Возврат`, гейт-отправитель → `[→] В работе`.
5. После доработки — повторная `$gates` проверка → forward handoff.

## Протокол

### Forward Handoff (прямая передача)

#### Шаг 1 — Предусловия
1. Убедиться, что `$gates` дал PASS или CONDITIONAL PASS.
2. Определить агента-получателя по реестру.
3. Собрать все артефакты из реестра передач.

#### Шаг 2 — Формирование конверта
1. Заполнить **все поля** шаблона (см. ниже).
2. Для CONDITIONAL PASS — явно указать gap'ы, которые передаются.
3. Сформулировать **чёткую задачу** для следующего агента (что делать, не как).

#### Шаг 3 — User sign-off
1. Показать конверт пользователю.
2. Ожидать явное **"Approved"**.
3. Не переходить без Approved. Без исключений.

#### Шаг 4 — Активация
1. Обновить `$board`: текущий гейт → `[✓] Завершён`, следующий → `[→] В работе`.
2. Добавить запись в лог передач.
3. Активировать следующего агента с полным контекстом.

### Reverse Handoff (возврат)

#### Шаг 1 — Формирование обратного конверта
1. Указать причину, gap'ы, что исправить, что не трогать.

#### Шаг 2 — User sign-off
1. Показать обратный конверт пользователю.
2. Ожидать Approved на возврат.

#### Шаг 3 — Активация
1. Обновить `$board`: текущий → `[↩] Возврат`, целевой → `[→] В работе`.
2. Добавить запись в лог передач (тип: Reverse).

## Валидация конверта (Quality Gate)

Handoff конверт считается полным если:

- [ ] Результат `$gates` приложен (PASS / CONDITIONAL PASS)
- [ ] Агент-отправитель и получатель указаны
- [ ] Все артефакты из реестра передач приложены
- [ ] Задача для следующего агента сформулирована конкретно
- [ ] Контекст передан (бренд, платформа, аудитория, ограничения)
- [ ] Ожидаемый deliverable указан
- [ ] Блокеры / gap'ы зафиксированы (или «Нет»)
- [ ] User Approved получен
- [ ] `$board` обновлён
- [ ] Запись в лог передач добавлена

## Интеграция с другими скилами

| Скил | Взаимодействие |
|------|----------------|
| `$gates` | **Предусловие:** handoff только после PASS/CONDITIONAL PASS |
| `$board` | **Обновление:** при каждом handoff — обновить статусы гейтов + лог событий |
| `$content-release-gate` | Финальный handoff → пакет публикации |
| Все скилы гейтов | Артефакты скилов = содержимое handoff конверта |

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Handoff без `$gates` | Непроверенные артефакты → проблемы на следующем гейте | Всегда `$gates` PASS → затем handoff |
| Handoff без Approved | Нарушение протокола, пользователь не в курсе | Всегда ждать явное Approved |
| «Сделай контент» вместо задачи | Агент не знает scope, делает лишнее или недостаточно | Конкретная задача: «Напиши текст по брифу IG-20260415-01, фокус на персону X» |
| Передача без артефактов | Следующий агент работает вслепую | Все артефакты из реестра приложены |
| Reverse handoff без конкретики | Агент не знает что исправлять | Конкретные gap'ы + что НЕ трогать |
| Не обновить `$board` | Доска рассинхронизирована, непонятно где мы | Обновлять `$board` при каждом handoff |
| Потеря контекста при передаче | Следующий агент переспрашивает | Бренд, платформа, аудитория, ограничения — в каждом конверте |

## Шаблон вывода

### Forward Handoff

```
### Handoff Envelope — [Агент From] → [Агент To]

**Тип:** Forward
**Режим:** [Full / Bugfix / Hotfix]
**Gate Check:** [PASS / CONDITIONAL PASS] (итерация [N])
**Дата:** [YYYY-MM-DD]

---

**Статус:** [Завершён / Условный]

**Артефакты:**
- [Артефакт 1 — описание / ссылка]
- [Артефакт 2 — описание / ссылка]
- [Артефакт 3 — описание / ссылка]

**Gap'ы (если CONDITIONAL):**
- [Gap 1 — что учесть]

---

**Задача для [Агент To]:**
[Чёткая конкретная формулировка: что сделать, scope, фокус]

**Контекст:**
- Бренд: [бренд]
- Платформа: [платформа]
- Аудитория: [персона из $audience-analysis]
- Ограничения: [ограничения]
- ToV: [тон из $brand-guidelines]

**Ожидаемый deliverable:**
[Что должен вернуть следующий агент — конкретный список]

**Блокеры:**
[Список блокеров или «Нет»]

---

→ Ожидаю **"Approved"** для перехода к **[Агент To]**
```

### Reverse Handoff

```
### Reverse Handoff — [Агент From] → [Агент To] (возврат)

**Тип:** Reverse
**Причина:** [FAIL на $gates / REQUIRES CHANGES от Reviewer / feedback пользователя]
**Дата:** [YYYY-MM-DD]

---

**Что исправить:**
| # | Gap / Замечание | Sev | Конкретное действие |
|---|----------------|-----|---------------------|
| 1 | [описание] | 🔴/🟡 | [что сделать] |
| 2 | [описание] | 🔴/🟡 | [что сделать] |

**Что НЕ трогать:**
- [Артефакт / секция, которая уже одобрена]

**Контекст:**
[Дополнительный контекст, если нужен]

---

→ После доработки — повторная проверка `$gates` (итерация [N+1])
→ Ожидаю **"Approved"** на возврат
```

### Лог передач

```
#### Лог передач — [Задача / ID]

| # | Дата | Тип | From → To | Gate Check | Статус | Gap'ы |
|---|------|-----|-----------|------------|--------|-------|
| 1 | [дата] | Forward | Conductor → Strategist | PASS | Завершён | — |
| 2 | [дата] | Forward | Strategist → Researcher | CONDITIONAL | Условный | [gap] |
| 3 | [дата] | Forward | Researcher → Copywriter | PASS | Завершён | — |
| 4 | [дата] | Reverse | Reviewer → Copywriter | FAIL | Возврат | [gap'ы] |
| 5 | [дата] | Forward | Copywriter → Reviewer | PASS | Завершён | — |
```
