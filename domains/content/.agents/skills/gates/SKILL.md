# Gates — Контроль гейтов пайплайна

## Когда использовать
- **Перед каждым переходом** между гейтами — проверка deliverable текущего гейта.
- **При возврате на доработку** — повторная проверка после исправлений.
- **При споре о готовности** — формальный арбитр: что сделано, а что нет.

> **Разграничение с `$board`:** `$gates` проверяет **качество и полноту** deliverable. `$board` отслеживает **статус и прогресс**. Gates отвечает на «готово ли?», Board — на «где мы сейчас?».

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Название гейта | ✅ | Какой гейт проверяем |
| Режим пайплайна | ✅ | Full / Bugfix / Hotfix — определяет набор гейтов и критерии |
| Deliverables гейта | ✅ | Артефакты, которые агент передаёт на проверку |
| Итерация | ⬚ | Если повторная проверка — номер итерации + gap'ы предыдущей |

## Реестр гейтов и deliverables

### Full Pipeline

| # | Гейт | Агент | Обязательные deliverables | Скилы |
|---|------|-------|--------------------------|-------|
| 1 | **Conductor** | Conductor | Режим определён, задача декомпозирована, доска создана | `$board` |
| 2 | **Strategist** | Strategist | Audience analysis, platform strategy, content calendar, competitor analysis | `$audience-analysis`, `$platform-strategy`, `$content-calendar`, `$competitor-content-analysis` |
| 3 | **Researcher** | Researcher | Topic research, source verification, data storytelling (если data-driven) | `$topic-research`, `$source-verification`, `$data-storytelling` |
| 4 | **Copywriter** | Copywriter | Финальный текст по content brief, CTA, headline | `$content-brief`, `$cta-optimization`, `$headline-formulas`, `$storytelling-framework` |
| 5 | **Visual Concept** | Visual Concept | Visual brief, image prompts, platform visual specs | `$visual-brief`, `$image-prompt-engineering`, `$platform-visual-specs` |
| 6 | **Reviewer** | Reviewer | Content review checklist, brand compliance, fact checking, readability | `$content-review-checklist`, `$brand-compliance`, `$fact-checking`, `$readability-scoring` |
| 7 | **Release Gate** | Conductor | Полный пакет для публикации, финальный чеклист | `$content-release-gate` |

### Bugfix Pipeline

| # | Гейт | Агент | Обязательные deliverables |
|---|------|-------|--------------------------|
| 1 | **Conductor** | Conductor | Проблема определена, режим Bugfix, доска создана |
| 2 | **Copywriter / Researcher** | Copywriter / Researcher | Исправленный контент / исправленные данные |
| 3 | **Reviewer** | Reviewer | Проверка исправлений: review checklist (фокус на исправленном) |
| 4 | **Release Gate** | Conductor | Финальная проверка исправлений |

### Hotfix Pipeline

| # | Гейт | Агент | Обязательные deliverables |
|---|------|-------|--------------------------|
| 1 | **Conductor** | Conductor | Проблема определена, режим Hotfix, blast radius ≈ 0 подтверждён |
| 2 | **Copywriter + Reviewer** | Copywriter | Исправление + self-check по mini-checklist |

## Критерии готовности по гейтам

### Conductor
- [ ] Режим пайплайна определён и обоснован
- [ ] Задача декомпозирована (что делаем, для чего, scope)
- [ ] Доска создана через `$board`
- [ ] Первый гейт активирован

### Strategist
- [ ] `$audience-analysis` — персоны определены, валидация пройдена
- [ ] `$platform-strategy` — платформы выбраны, частота зафиксирована
- [ ] `$content-calendar` — календарь заполнен, баланс-проверка пройдена
- [ ] `$competitor-content-analysis` — gaps выявлены, рекомендации сформулированы
- [ ] Все артефакты согласованы между собой (персоны ↔ платформы ↔ календарь)

### Researcher
- [ ] Тема исследована, ключевые факты собраны (`$topic-research`)
- [ ] Источники верифицированы (`$source-verification`)
- [ ] Data storytelling подготовлен (если data-driven контент)
- [ ] Факты готовы к передаче Copywriter'у

### Copywriter
- [ ] Текст написан по `$content-brief`
- [ ] Тема, тезисы, персона соответствуют брифу
- [ ] CTA оптимизирован (`$cta-optimization`)
- [ ] Объём соответствует платформе
- [ ] ToV соответствует `$brand-guidelines`

### Visual Concept
- [ ] Visual brief создан (`$visual-brief`)
- [ ] AI-промпты подготовлены (если AI-генерация)
- [ ] Размеры соответствуют платформе (`$platform-visual-specs`)
- [ ] Стиль соответствует `$brand-guidelines`

### Reviewer
- [ ] `$content-review-checklist` — пройден полностью, решение APPROVED
- [ ] `$brand-compliance` — статус COMPLIANT
- [ ] `$fact-checking` — статус PASS / VERIFIED
- [ ] `$readability-scoring` — в допустимом диапазоне (если применимо)
- [ ] Все замечания исправлены (если был REQUIRES CHANGES)

### Release Gate
- [ ] Все предыдущие гейты пройдены (статус `[✓]` на `$board`)
- [ ] `$content-release-gate` чеклист пройден — 0 блокеров
- [ ] Пакет публикации сформирован
- [ ] User sign-off получен

## Severity критериев

| Severity | Определение | Влияние на решение |
|----------|-------------|-------------------|
| 🔴 **Blocker** | Обязательный deliverable отсутствует или критически неполный | FAIL — возврат на доработку |
| 🟡 **Gap** | Deliverable есть, но неполный или с замечаниями | CONDITIONAL PASS — можно передать, но gap'ы зафиксированы |
| 🔵 **Note** | Мелкое замечание, не влияет на переход | PASS — замечание передаётся следующему гейту как контекст |

## Решения

| Решение | Условие | Действие |
|---------|---------|----------|
| **PASS** | 0 Blocker + 0 Gap | → Запросить user Approved → `$handoff` → следующий гейт |
| **CONDITIONAL PASS** | 0 Blocker + 1-2 Gap | → Зафиксировать gap'ы → Запросить user Approved с пометкой gap'ов → `$handoff` с пометкой |
| **FAIL** | ≥ 1 Blocker | → Вернуть агенту с конкретным списком gap'ов → Не запрашивать Approved |

> **Абсолютное правило:** Никогда не пропускать гейты, даже если задача кажется простой. FAIL → доработка → повторная проверка. Без исключений.

## Протокол

### Шаг 1 — Определение контекста
1. Определить какой гейт проверяем.
2. Определить режим пайплайна (Full / Bugfix / Hotfix).
3. Загрузить критерии готовности для этого гейта из реестра.
4. Если повторная проверка — загрузить gap'ы предыдущей итерации.

### Шаг 2 — Проверка deliverables
1. Для каждого критерия готовности — проверить: выполнен ✅ / не выполнен ❌ / N/A.
2. Для каждого ❌ — присвоить severity (🔴 Blocker / 🟡 Gap / 🔵 Note).
3. Для каждого ❌ — описать конкретный gap: что именно не хватает / что исправить.

### Шаг 3 — Принятие решения
1. Подсчитать severity.
2. Применить правила решений (PASS / CONDITIONAL PASS / FAIL).
3. Добавить запись в лог проверок.

### Шаг 4 — Действие по результату
- **PASS:** показать пользователю → запросить Approved → `$handoff` → обновить `$board`.
- **CONDITIONAL PASS:** показать gap'ы → запросить Approved с подтверждением gap'ов → `$handoff` с пометкой → обновить `$board`.
- **FAIL:** показать gap'ы → вернуть агенту → обновить `$board` (статус `[↩] Возврат`).

## Валидация (Quality Gate)

Проверка гейта считается завершённой если:

- [ ] Все критерии готовности пройдены (✅ / ❌ / N/A — нет пустых)
- [ ] Для каждого ❌ указаны: severity, описание gap'а, что исправить
- [ ] Решение соответствует правилам (не субъективное)
- [ ] Лог проверок обновлён
- [ ] При PASS/CONDITIONAL PASS — user Approved получен
- [ ] `$board` обновлён

## Интеграция с другими скилами

| Скил | Взаимодействие |
|------|----------------|
| `$board` | **Обновление:** статус гейта по результату проверки |
| `$handoff` | **Потребитель:** при PASS — передача через handoff с приложением gate check |
| `$content-release-gate` | **Финальный гейт:** Release Gate = последняя проверка `$gates` |
| Все скилы гейтов | **Источник критериев:** deliverables скилов = критерии проверки |

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Пропуск гейта «потому что просто» | Ошибки проходят незамеченными | Никогда не пропускать. Без исключений |
| PASS без проверки всех критериев | Незамеченные gap'ы | Каждый критерий = ✅ / ❌ / N/A |
| «В целом ок» вместо конкретики | Агент не знает что исправлять | Конкретный gap: «отсутствует CTA в тексте» |
| FAIL без описания gap'ов | Агент возвращается и не знает что делать | Каждый FAIL = список конкретных доработок |
| Approved без user sign-off | Нарушение протокола пайплайна | Всегда явное Approved от пользователя |
| Повторная проверка без сверки с предыдущими gap'ами | Gap мог быть не исправлен | Загрузить gap'ы предыдущей итерации, проверить каждый |
| Все критерии одинаковой важности | Мелочь блокирует, критичное проходит | Severity: Blocker / Gap / Note |

| # | Критерий | Sev | Статус | Gap / Комментарий |

---

#### Сводка

| Severity | Количество | Пункты |
|----------|-----------|--------|
| 🔴 Blocker | [N] | #[список] |
| 🟡 Gap | [N] | #[список] |
| 🔵 Note | [N] | #[список] |

### Решение: [ PASS / CONDITIONAL PASS / FAIL ]

- [Артефакт 2]
