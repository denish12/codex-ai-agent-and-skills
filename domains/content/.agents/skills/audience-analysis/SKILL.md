# Audience Analysis — Анализ целевой аудитории

## Когда использовать
- При старте контент-стратегии — до `$content-calendar`, `$platform-strategy`, `$tone-of-voice`.
- При выходе на новый рынок / запуске нового продукта / ребрендинге.
- При заметном расхождении метрик вовлечённости с ожиданиями (пересмотр аудитории).

## Вход
Агент **обязан** получить от пользователя или предыдущего гейта:

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Продукт / бренд | ✅ | Что продвигаем |
| Ниша / индустрия | ✅ | В какой отрасли работаем |
| Гео | ✅ | Целевые рынки (страны / регионы) |
| Бизнес-модель | ✅ | B2B / B2C / B2B2C / DTC |
| Текущие каналы | ⬚ | Где уже есть присутствие |
| Существующая аналитика | ⬚ | GA, соцсети, CRM-данные, опросы |
| Гипотезы по аудитории | ⬚ | Кого пользователь считает ЦА |

> Если обязательные поля не предоставлены — **запросить у пользователя** до начала анализа. Не додумывать.

## Источники данных
Агент должен опираться на проверяемые данные, а не на догадки:

1. **Веб-поиск** — отраслевые отчёты, демографические исследования, статистика платформ.
2. **Конкуренты** — анализ подписчиков, комментариев, отзывов конкурентов (пересечение с `$competitor-content-analysis`).
3. **Платформенная аналитика** — если пользователь предоставил доступ к GA / Meta Insights / Telegram Stats.
4. **Отзывы и форумы** — Reddit, Trustpilot, тематические сообщества, YouTube-комменты.
5. **Данные из CRM / опросов** — если предоставлены пользователем.

> Для каждой персоны указать минимум 2 источника, подтверждающих её существование.

## Протокол

### Шаг 0 — Сбор контекста
Проверить наличие всех обязательных входных данных. Если чего-то не хватает — запросить у пользователя.

### Шаг 1 — Сегментация
Определить **2-5 сегментов** аудитории на основе входных данных и исследования. Критерии сегментации зависят от бизнес-модели:
- **B2C:** демографика + поведение + мотивация.
- **B2B:** размер компании + роль ЛПР + этап зрелости.
- **B2B2C:** оба слоя (бизнес-клиент + конечный потребитель).

### Шаг 2 — Построение персон
Для каждого сегмента — создать персону с полным профилем (см. шаблон вывода).

### Шаг 3 — Карта болей и задач (Jobs-to-be-Done)
Для каждой персоны:
- **Функциональные задачи** — что человек пытается сделать.
- **Эмоциональные задачи** — как хочет себя чувствовать.
- **Социальные задачи** — как хочет выглядеть в глазах окружающих.
- **Боли** — что мешает выполнить задачи.
- **Текущие альтернативы** — как решает проблему сейчас.

### Шаг 4 — Триггеры и барьеры
- **Триггеры** — события / обстоятельства, которые подталкивают к действию.
- **Барьеры** — что мешает конвертироваться (цена, доверие, сложность).

### Шаг 5 — Платформы и форматы
- Где каждая персона проводит время.
- Какие форматы контента потребляет.
- В какое время наиболее активна.

### Шаг 6 — Тон и язык
- Какой тон резонирует (формальный / дружеский / экспертный / провокационный).
- Какие слова и фразы использует сама аудитория (язык клиента).
- Какие слова-триггеры работают / какие отталкивают.

### Шаг 7 — Приоритизация
Ранжировать персоны по критериям:
- Размер сегмента (охват).
- Платёжеспособность / LTV.
- Доступность через контент-каналы.
- Стратегическая важность для бизнеса.

## Валидация (Quality Gate)
Анализ считается завершённым **только** если все пункты выполнены:

- [ ] Все обязательные входные данные получены
- [ ] Определено 2-5 персон (не больше, не меньше)
- [ ] Персоны различаются между собой (нет дублирования сегментов)
- [ ] Для каждой персоны указано ≥ 2 источника данных
- [ ] Боли и триггеры конкретны и actionable (не «хочет лучшего», а «не успевает создавать контент из-за ручного планирования»)
- [ ] Указаны платформы и форматы для каждой персоны
- [ ] Определён приоритетный сегмент с обоснованием
- [ ] Нет внутренних противоречий между персонами

> Если валидация не пройдена — доработать до повторного прохождения, а не передавать дальше.

## Handoff
Результат этого скила является входом для:
- `$platform-strategy` — выбор каналов на основе платформенных предпочтений персон.
- `$tone-of-voice` — построение тона на основе языка и тона каждого сегмента.
- `$content-brief` — выбор персоны для конкретной единицы контента.
- `$content-calendar` — планирование тем под боли и триггеры конкретных персон.

При передаче — использовать `$handoff` с указанием всех персон и приоритетного сегмента.

## Anti-patterns
| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Персоны без источников | Фантазия, не анализ | Каждая персона подкреплена данными |
| «Все 18-65» | Бесполезная сегментация | Узкие, различимые сегменты |
| Только демографика | Люди одного возраста ведут себя по-разному | Добавить поведение, мотивацию, JTBD |
| Копирование конкурентов | Нет дифференциации | Найти незанятые ниши аудитории |
| 10+ персон | Невозможно создавать контент для всех | Максимум 5, приоритизировать |
| Абстрактные боли | «Хочет лучшего» — не actionable | Конкретные проблемы с контекстом |
