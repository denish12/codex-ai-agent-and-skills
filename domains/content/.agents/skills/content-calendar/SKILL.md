# Content Calendar — Контент-календарь

## Когда использовать
- При **стратегическом планировании** — еженедельно, ежемесячно, квартально или под кампанию.
- При **масштабировании контента** — переход от хаотичных публикаций к системе.
- При **пересмотре стратегии** — ребалансировка после анализа метрик или смены приоритетов.
- После **`$audience-analysis`**, **`$platform-strategy`**, **`$competitor-content-analysis`** — календарь собирает их выводы в план действий.

## Вход

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Горизонт планирования | ✅ | Неделя / месяц / квартал / кампания (с датами) |
| Персоны ЦА | ✅ | Из `$audience-analysis` — для распределения по постам |
| Платформы и частота | ✅ | Из `$platform-strategy` — где и сколько публикуем |
| Бренд-гайдлайны | ✅ | Из `$brand-guidelines` — ToV, визуальный стиль |
| Бизнес-цели периода | ✅ | Что бизнес хочет от контента в этом периоде (запуск, лиды, awareness) |
| Инфоповоды | ⬚ | Праздники, отраслевые события, запуски продуктов, сезоны |
| Рекомендации из анализа конкурентов | ⬚ | Из `$competitor-content-analysis` — gaps и возможности |
| Результаты `$trend-research` | ⬚ | Актуальные тренды для встраивания |
| Рубрики | ⬚ | Если уже определены — повторяющиеся форматы |
| Предыдущий календарь + метрики | ⬚ | Для итерации: что сработало, что нет |

> Если персоны или платформы не предоставлены — **запросить у пользователя** или направить на `$audience-analysis` / `$platform-strategy`.

## Ключевые определения

### Типы контента (контент-микс)

| Тип | Описание | Пример |
|-----|----------|--------|
| Образовательный | How-to, tips, разборы, гайды | «5 ошибок email-маркетинга» |
| Продающий | Офферы, кейсы, отзывы, демо | «Клиент увеличил конверсию на 40%» |
| Развлекательный | Мемы, тренды, интерактив | Мем про дедлайны + опрос |
| Имиджевый | Миссия, команда, за кулисами | «Как мы проводим ретро» |
| Комьюнити | UGC, вопросы, ответы, челленджи | «Покажите ваш workspace» |

### Этапы воронки

| Этап | Цель | KPI |
|------|------|-----|
| **TOFU** | Awareness — привлечь новую аудиторию | Охват, подписки |
| **MOFU** | Consideration — создать доверие | Вовлечённость, сохранения, время чтения |
| **BOFU** | Decision — конвертировать | Клики, лиды, продажи |

## Протокол

### Шаг 1 — Каркас периода
1. Зафиксировать **горизонт** (даты начала и конца).
2. Выписать **инфоповоды периода**: праздники, отраслевые события, запуски, сезонные триггеры.
3. Определить **бизнес-приоритеты**: что важнее в этом периоде (awareness / лиды / продажи).

### Шаг 2 — Рубрики
1. Определить **повторяющиеся рубрики** (если есть или нужны):
   - Фиксированный день + формат (напр.: понедельник = Tip, среда = кейс, пятница = мем).
   - Серии (напр.: «Мифы отрасли», «Инструмент недели»).
2. Рубрики заполняют ~40-60% слотов. Остальное — ситуативный контент.

### Шаг 3 — Распределение тем
Для каждого слота в календаре определить:
1. **Тема** — конкретный угол (не «маркетинг», а «почему UTM-метки теряются в email»).
2. **Персона** — для какой персоны из `$audience-analysis`.
3. **Этап воронки** — TOFU / MOFU / BOFU.
4. **Тип контента** — образовательный / продающий / развлекательный / имиджевый / комьюнити.
5. **Формат** — пост / карусель / reel / статья / email / thread / сторис.
6. **Платформа** — основной канал.

Источники тем:
- Боли и триггеры персон (из `$audience-analysis`).
- Gaps конкурентов (из `$competitor-content-analysis`).
- Тренды (из `$trend-research`).
- Инфоповоды периода.
- Рубрики.

### Шаг 4 — Кросс-постинг
Для каждой единицы контента определить:
1. Подходит ли для адаптации на другие платформы?
2. Какие изменения нужны (формат, объём, CTA, визуал)?
3. Добавить строки кросс-постинга в календарь (с пометкой `↳ адаптация`).

### Шаг 5 — Дедлайны производства
Для каждой единицы контента — обратный отсчёт от даты публикации:

| Этап | Смещение | Ответственный |
|------|----------|---------------|
| Бриф готов | Публикация − N дней | Strategist |
| Текст готов | Публикация − N дней | Copywriter |
| Визуал готов | Публикация − N дней | Visual Concept |
| Ревью пройден | Публикация − N дней | Reviewer |
| Готов к публикации | Публикация − 1 день | — |

> Конкретные смещения зависят от объёма и сложности. Согласовать с пользователем при первом планировании, затем использовать как стандарт.

### Шаг 6 — Баланс-проверка
Проверить календарь по 4 осям:

1. **Контент-микс** — пропорции по типам:
   - Рекомендуемый баланс (адаптировать под бизнес): Образовательный 30-40% · Продающий 15-25% · Развлекательный 15-20% · Имиджевый 10-15% · Комьюнити 10-15%.
   - Если > 40% продающего — календарь «давит» на аудиторию.

2. **Воронка** — покрытие этапов:
   - TOFU 40-50% · MOFU 30-35% · BOFU 15-25%.
   - Не должно быть недели без TOFU (нет притока новой аудитории).

3. **Персоны** — равномерное покрытие:
   - Приоритетная персона → больше контента, но остальные не должны быть забыты.

4. **Форматы** — разнообразие:
   - Не более 50% одного формата (все посты — текст без визуала = скучно).

### Шаг 7 — Присвоение ID
Каждой единице контента присвоить уникальный ID:
```
[Платформа]-[YYYYMMDD]-[порядковый номер]
Пример: IG-20260415-01, TG-20260415-02
```
Этот ID используется в `$content-brief` для привязки брифа к единице календаря.

## Валидация (Quality Gate)

Календарь считается готовым если:

- [ ] Все слоты заполнены (нет пустых дат в рамках частоты из `$platform-strategy`)
- [ ] Каждая единица имеет: тему, персону, воронку, тип, формат, платформу
- [ ] Каждая единица имеет уникальный ID
- [ ] Контент-микс в допустимых пропорциях (нет перекоса > 40% одного типа)
- [ ] Воронка сбалансирована (есть TOFU + MOFU + BOFU каждую неделю)
- [ ] Все персоны покрыты (приоритетная — чаще, но остальные не забыты)
- [ ] Инфоповоды учтены (праздники, запуски, сезоны)
- [ ] Кросс-постинг размечен (адаптации с пометкой)
- [ ] Дедлайны производства проставлены
- [ ] Баланс-проверка пройдена по 4 осям
- [ ] Согласован с пользователем

## Handoff

| Итог | Маршрут |
|------|---------|
| Календарь согласован | → `$content-brief` для каждой единицы контента (по ID) |
| Нужно исследование перед брифом | → `$topic-research` / `$trend-research` → затем `$content-brief` |
| Нужен пересмотр стратегии | → обратно к `$audience-analysis` / `$platform-strategy` |

При передаче — использовать `$handoff` с приложением календаря и баланс-сводки.

## Интеграция с другими скилами

| Скил | Взаимодействие |
|------|----------------|
| `$audience-analysis` | **Источник:** персоны, боли, триггеры → темы |
| `$platform-strategy` | **Источник:** платформы, частота, форматы, время |
| `$competitor-content-analysis` | **Источник:** gaps → темы и форматы |
| `$trend-research` | **Источник:** тренды → ситуативный контент |
| `$tone-of-voice` | **Источник:** тон для каждой персоны |
| `$content-brief` | **Потребитель:** каждая единица календаря → бриф по ID |
| `$board` | Обновить прогресс по производству контента |

## Anti-patterns

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Всё продающее | Аудитория устаёт, отписки | Продающий контент ≤ 25% |
| Нет TOFU | Нет притока новой аудитории | TOFU 40-50% каждую неделю |
| Темы без угла | «SEO» — это не тема | Конкретный угол: «Почему alt-теги важнее title» |
| Нет персоны у поста | Контент «для всех» = для никого | Каждый пост — конкретная персона |
| Календарь без буфера | Одна болезнь — и всё сломалось | 1-2 слота в неделю = буфер / ситуативный |
| Забыты праздники | Конкуренты поздравляют, вы молчите | Выписать инфоповоды до планирования |
| Нет кросс-постинга | Контент делается 1 раз для 1 платформы | Один контент → 2-3 адаптации |
| Одинаковый формат | Все посты — текст без визуала | Чередовать форматы, ≤ 50% одного типа |
| Без дедлайнов | Контент не готов к дате публикации | Обратный отсчёт от даты публикации |
