# Skill: TDD Workflow

## Цель
Гарантировать надёжность изменений и уверенный рефакторинг.

## Правила
- Тесты пишутся ДО кода (RED → GREEN → REFACTOR).
- Тесты проверяют поведение, а не детали реализации.
- Минимум: unit + integration. E2E — для критичных user flows (по решению дирижёра/архитектора).

## Выход
- Набор тестов на фичу + реализация + отчёт о покрытии/рисках.

## Алгоритм
1) Сформулируй user journey (роль → действие → ценность).
2) Сгенерируй тест-кейсы (happy path + edge cases + error paths).
3) Запусти тесты (должны падать).
4) Реализуй минимальный код до “зелени”.
5) Рефакторинг (читаемость, DRY, архитектурные границы).
6) Проверка покрытия (порог проекта) и отсутствие флейков.

## Ошибки, которых избегать
- Тесты на внутреннее состояние вместо внешнего поведения
- Хрупкие селекторы
- Зависимость тестов друг от друга
- Пропуск error-path тестов
