# Skill: PRD (Product Requirements Document)

## Цель
Создать PRD, по которому UX/UI, архитектор, разработчик, ревьюер и тестировщик могут работать без “догадок”.

## Выход
PRD в следующей структуре (вставкой в ответ или как файл в репо):

### 1) Vision & Goals
- Проблема, которую решаем
- Целевая аудитория
- Ценность/уникальность
- Метрики успеха

### 2) Scope
- MVP (In)
- Out of scope
- Допущения и ограничения

### 3) Roles & Permissions (если применимо)
- Роли пользователей
- Что кому можно/нельзя

### 4) User Stories / Scenarios
- Список основных сценариев (вертикальными “потоками”)

### 5) Acceptance Criteria (обязательно проверяемые)
Для каждого сценария:
- Given / When / Then (или чек-лист)
- Позитивные + негативные кейсы
- Ошибки/валидация/пустые состояния

### 6) Non-Functional Requirements
- Security
- Performance
- Reliability
- Observability
- Accessibility (минимум)
- Localization (если нужно)

### 7) Risks & Open Questions
- Риски (технические, продуктовые, юридические)
- Открытые вопросы и кому адресованы

## Качество PRD (чек-лист)
- MVP можно реализовать за 1–3 итерации
- У каждого сценария есть критерии приёмки
- Есть out-of-scope (чтобы не расползалось)
- NFR не забыты
- Вопросы и допущения явно помечены
- PRD понятно без устного объяснения (можно отдать UX/архитектору/разработчику и они поймут, что делать)
